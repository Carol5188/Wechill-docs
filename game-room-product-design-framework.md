# 语聊房新增游戏房产品设计框架

生成日期：2026-04-22  
输入依据：

- `BytesunGame（语聊房模式）对接文档1.0.7 (2).pdf`
- 当前语聊房 / CP / 房间玩法相关文档
- 现有产品方向：在语聊房基础上新增游戏房类别，支持飞行棋 Ludo、UNO、你画我猜等轻社交游戏

## 1. 产品定位

游戏房不是单独的游戏 App，也不是纯三方游戏大厅，而是“语聊房 + 三方 H5 游戏”的复合房间类型。

核心目标：

1. 给语聊房增加明确的游戏场景分类，提高房间进入理由。
2. 通过 Ludo、UNO、你画我猜等轻游戏增强陌生人破冰、房间留存和互动时长。
3. 复用现有语聊房的用户、麦位、IM、礼物、钱包、风控、房间推荐和运营体系。
4. 通过 Bytesun 承接游戏玩法、牌局/局内状态、结算逻辑，降低自研游戏成本。

## 2. Bytesun 对接文档关键约束

### 2.1 接入方式

- 商户侧调用 Bytesun 游戏信息列表接口，拿到游戏 Zip 包下载地址或 URL 地址、游戏版本号。
- 客户端按版本号处理游戏包更新，Zip 包可下载并存储到 App 本地。
- 用户访问解压后的本地路径或 URL，进入游戏 H5 界面。
- 游戏内下注、结算时，Bytesun 服务端调用我方服务端 API 修改用户游戏币。
- 游戏结束后回到我方 App / 语聊房。
- Bytesun 游戏离线 Zip 包默认是 OSS 源站；为体验建议我方配置 CDN 回源，缓存策略建议大于 3 天，并做预热。

### 2.2 前端 JSBridge

客户端需要支持 Bytesun H5 调用 App，也需要 App 主动调用游戏 H5。

H5 调 App：

- `getConfig`：获取商户、用户、房间、语言、角色、节点等配置。
- `destroy`：非语聊房模式下游戏主动关闭 WebView；语聊房模式通常不依赖。
- `gameRecharge`：余额不足或点击金币图标时，通知 App 打开商城。
- `gameLoaded`：游戏加载完毕，App 可关闭自己的加载动画。
- `sendGameAction`：游戏主动通知 App，包括点击用户、游戏开始/结束、上下座、座位同步、踢人、画猜聊天、游戏参数、RTC 麦克风/扬声器同步等。

App 调 H5：

- `walletUpdate`：App 充值或货币变化后通知游戏刷新余额。
- `gameActionUpdate`：App 通知游戏座位操作、身份变更、踢人结果、聊天内容同步、语聊房最小化状态等。
- `soundUpdate`：设置游戏背景音乐和音效开关。

### 2.3 `getConfig` 关键字段

| 字段 | 设计口径 |
| --- | --- |
| `appChannel` | Bytesun 提供，后台配置。 |
| `appId` | Bytesun 提供的商户 ID。 |
| `userId` | 我方用户 ID。 |
| `code` | 我方生成的一次性认证令牌，Bytesun 用它向我方服务端换 `ss_token`。必须唯一且一次性。 |
| `roomId` | 我方语聊房 / 游戏房房间 ID。 |
| `gameRoomId` | 文档要求传空字符串，保留字段。 |
| `gameMode` | 语聊房场景固定传 `"3"`。 |
| `language` | 语言值，英文 `2`，阿语 `7`，中文 `0`。 |
| `gameConfig.sceneMode` | 场馆级别，默认 0。 |
| `gameConfig.currencyIcon` | 游戏货币图标，需外网可访问，建议 60*60。 |
| `gsp` | 游戏服务器节点；MENA 优先考虑 `201` 迪拜 AWS，必要时降级新加坡 `101`。 |
| `role` | `0` 正常用户，可游戏可观战；`1` 游客，仅观战；`2` 主持人，可游戏可观战可开启游戏。 |

### 2.4 服务端必须提供的 API

我方需要给 Bytesun 配置测试和正式服务端接口：

| 接口 | 用途 | 关键要求 |
| --- | --- | --- |
| `/v1/api/get_sstoken` | Bytesun 使用一次性 `code` 换长期 `ss_token`。 | `code` 必须一次性消费，再次使用返回错误。 |
| `/v1/api/get_user_info` | Bytesun 查询用户昵称、头像、余额、用户类型。 | 返回 `balance`，可扩展多币种 `balance_list`。 |
| `/v1/api/update_sstoken` | `ss_token` 过期时刷新。 | 如果我方 `ss_token` 非长期有效，需要实现。 |
| `/v1/api/change_balance` | 下注、结算、退款时修改用户货币。 | 必须做单用户并发锁；`order_id` 唯一；余额不足错误码为 `1008`；成功 code 才能为 `0`。 |
| 游戏状态上报接口 | Bytesun 上报 `game_start`、`game_settle`。 | 用于局记录、数据看板、风控和用户战绩。 |

安全：

- 请求双方用 `signature_nonce + AppKey + timestamp` 做 MD5 签名。
- `signature` 15 秒内有效。
- `signature_nonce` 15 秒内不能重复，防重放。
- AppKey 必须严格保密。

### 2.5 Bytesun 游戏信息接口

Bytesun 提供：

- `/v1/api/one_game_info`：获取单个游戏信息。
- `/v1/api/gamelist`：获取游戏信息列表。

返回字段包括：

- `game_id`
- `name`
- `preview_url`
- `game_version`
- `download_url`
- `game_mode`
- `game_orientation`：1 竖屏，2 横屏。
- `safe_height`
- `venue_level`

注意：文档字段说明中 `game_list_type` 写“2 游戏”，示例请求写的是 `3`，上线前需要和 Bytesun 确认真实取值。

### 2.6 特殊游戏能力

- Ludo / UNO：文档说明 `hideLobby=true` 目前支持 `DominoPlus`、`ludoPlus`、`unoPlus`。开启后，游戏启动若不是重连，会停留在 loading 页，等待我方服务端调用“快速开始游戏 API”直接开启游戏。
- 你画我猜 / 画猜类：游戏通过 `sendGameAction type=22` 把消息发到 App 聊天区；App 通过 `gameActionUpdate type=2014` 把聊天区内容同步回游戏。
- 需要游戏内 RTC 的游戏，例如狼人杀、谁是卧底：URL 可加 `isGameRTC=true`，Android 需要授权音频捕获；普通 Ludo / UNO 建议继续使用我方语聊房 RTC。
- game_id 以 3 开头的全屏语聊房模式：需要追加安全区参数 `game_margin_top`、`game_margin_bottom`、`game_margin_standard`，避免游戏操作区被 App 麦位、顶部栏或底部操作栏遮挡。

## 3. 产品信息架构

### 3.1 房间类别新增

在原有语聊房房间类型中新增 `游戏房`。

建议房间类型：

1. 语聊房
2. 主播房
3. 派对房
4. CP 房 / 关系房
5. 游戏房

游戏房下再按游戏分类：

- 桌游：Ludo / Domino
- 卡牌：UNO
- 画猜：你画我猜
- 推理语音：狼人杀、谁是卧底，后续版本
- 休闲小游戏：贪吃、Slots 等视地区合规情况开放

### 3.2 游戏房关键字段

| 字段 | 说明 |
| --- | --- |
| `room_type` | 固定为 `game_room`。 |
| `voice_room_id` | 我方房间 ID，也是传给 Bytesun 的 `roomId`。 |
| `game_id` | Bytesun 游戏 ID。 |
| `game_name` | 游戏名，如 Ludo、UNO、你画我猜。 |
| `game_category` | 桌游、卡牌、画猜、推理等。 |

| `game_orientation` | 竖屏 / 横屏。 |

| `min_players` | 最低开局人数。 |
| `max_players` | 当前游戏最大人数，可由 Bytesun `type=30 peopleNum` 同步。 |

## 4. 用户角色

| 角色 | Bytesun role | 权限 |
| --- | --- | --- |
| 房主 / 主持人 | `2` | 可参与、可观战、可开启游戏、可踢人、可调整房间配置。 |
| 游戏玩家 | `0` | 可上座、准备、参与游戏、发言、送礼。 |

## 5. 前台入口设计

### 5.1 首页 / 房间列表

新增一级入口：

- Tab：`游戏`
- 分类筛选：全部、Ludo、UNO、你画我猜、热门、好友在玩
- 快捷按钮：`随机匹配`
- 房间卡片展示：
  - 游戏 icon
  - 房间名
  - 游戏名
  - 当前人数 / 最大人数
  - 状态：等待中、准备中、游戏中、可观战
  - 门票档位 / 免费
  - 房主头像
  - 语言 / 国家
  - 是否可随机匹配

### 5.2 语聊房内游戏入口

普通语聊房也可以挂游戏入口：

- 房主点击 `开启游戏`
- 选择 Ludo / UNO / 你画我猜
- 当前语聊房临时切换为游戏房状态
- App 打开游戏 WebView 浮层或全屏游戏层
- 房内麦位和游戏座位进行同步

这个模式适合已有房间变成游戏局，不一定进入公开游戏房列表，取决于房主配置。

### 5.3 游戏房创建页

创建项：

- 选择游戏
- 房间名称
- 语言 / 地区
- 房间公开性
- 最大人数
- 观战开关
- 随机匹配开关
- 门票 / 底注档位
- 是否允许中途加入
- 是否允许游戏中观战
- 是否开启礼物 / 红包 / CP 任务联动

默认建议：

- Ludo：2-4 人，允许观战，允许随机匹配。
- UNO：2-4 人或 2-6 人，以 Bytesun 当前支持人数为准，允许观战，允许随机匹配。
- 你画我猜：4-8 人，允许观战，强依赖聊天同步和内容审核。

## 6. 核心用户链路

### 6.1 创建游戏房

1. 房主进入创建页。
2. 选择游戏、语言、人数、门票、是否公开。
3. 我方创建 `game_room` 类型语聊房，生成 `voice_room_id`。
4. 服务端生成一次性 `code`。
5. 客户端打开 Bytesun H5。
6. H5 调 `getConfig`，App 返回 `roomId=voice_room_id`、`gameMode=3`、`role=2`。
7. 收到 `gameLoaded` 后关闭 App loading。
8. 收到 `type=20` 语聊房游戏准备完成后，App 可按需自动同步房主上座。
9. 房主邀请用户或等待随机匹配补人。

### 6.2 加入游戏房

1. 用户从游戏 Tab、房间推荐、好友邀请、随机匹配进入房间。
2. 我方判断房间状态：
   - 等待中 / 准备中：可作为玩家加入。
   - 游戏中：默认以观众身份加入，除非游戏支持中途加入。
3. 客户端打开对应游戏 H5。
4. H5 调 `getConfig`，App 返回用户 `role`：
   - 可参与：`0`
   - 仅观战：`1`
5. H5 同步座位信息 `type=16`。
6. App 检查游戏座位和我方麦位 / 玩家位是否一致。
7. 不一致时，App 可通过 `gameActionUpdate type=4` 以我方房间座位为准同步给游戏。

### 6.3 开局与结算

1. 游戏通知 App `type=13`：游戏开始。
2. Bytesun 服务端上报 `game_start`。
3. 游戏内下注时，Bytesun 调我方 `/change_balance` 扣币。
4. 余额不足时，游戏 H5 调 `gameRecharge`，App 打开商城。
5. 用户充值成功后，App 调 `walletUpdate` 通知游戏刷新余额。
6. 游戏结束时，H5 通知 App `type=14`。
7. Bytesun 服务端上报 `game_settle`。
8. 结算奖励时，Bytesun 调我方 `/change_balance` 加币或退款。
9. App 展示局后结果：排名、奖励、再来一局、邀请好友、返回房间。

### 6.4 你画我猜聊天链路

1. 游戏需要把猜测消息发到 App 聊天区时，H5 调 `sendGameAction type=22`。
2. App 根据 `isSystem`、`isPlayer`、`userId`、`nickName`、`pos`、`content` 展示聊天消息。
3. App 用户在聊天区输入猜测时，App 调 `gameActionUpdate type=2014` 把 `userId` 和 `content` 同步给游戏。
4. 我方必须对聊天内容做敏感词、辱骂、广告、涉政涉黄等审核；必要时只同步审核通过的内容给游戏。

## 7. 随机匹配要不要我方做

结论：建议我方做随机匹配，而且从 V2 开始做成平台能力。

原因：

1. 游戏房是我方语聊房的一种房间类别，用户先进入的是我方社交场，不只是 Bytesun 游戏大厅。
2. 我方需要控制推荐、分区、语言、房间状态、麦位、风控、钱包、礼物、CP、活动任务和数据归因。
3. Bytesun 文档只负责游戏 H5、局内座位、开局、结算、余额变更，不覆盖我方房间推荐和跨房匹配。
4. `hideLobby=true` 虽然支持 Ludo / UNO，但文档明确说要由 App 端服务器调用“快速开始游戏 API”直接开启游戏；这个 API 在 1.0.7 文档中未给出定义，需要向 Bytesun 补充确认。
5. 如果完全依赖三方大厅，产品会失去“游戏房类别”的运营能力，也难以保证用户等待体验和回流链路。

推荐边界：

- 我方负责：随机匹配、等待队列、房间创建/路由、语言/地区/风险过滤、座位预占、超时降级、邀请好友、数据看板。
- Bytesun 负责：游戏 H5、游戏规则、局内状态、牌局/棋局逻辑、结算结果、局内座位事件。
- 双方协作：Ludo / UNO 的 `hideLobby=true` 快速开局，需要 Bytesun 提供明确 server API；在 API 未确认前，V1 不强依赖隐藏大厅。

## 8. 随机匹配产品方案

### 8.1 匹配入口

入口建议：

- 游戏 Tab 顶部：`随机匹配`
- 游戏卡片：`快速开始`
- 房间详情：`缺人中，加入本局`
- 结算页：`再来一局`
- 好友邀请卡：`一起玩`

### 8.2 匹配对象

随机匹配不是直接把用户丢进 Bytesun 大厅，而是：

1. 进入我方匹配池。
2. 匹配到已有等待中游戏房，或创建新的游戏房。
3. 预占玩家位。
4. 进入我方语聊房壳。
5. 打开 Bytesun H5。
6. 同步上座、准备、开局。

### 8.3 匹配维度

第一优先级，必须一致：

- `game_id`
- 语言
- 地区 / gsp 节点
- 房间状态：等待中或准备中
- 用户身份：非封禁、非高风险
- App 版本 / 游戏包版本可用

第二优先级，建议一致：

- 门票档位 / 底注
- 最大人数
- 是否允许观战
- 是否允许陌生人
- 用户等级或新老用户层级

第三优先级，可逐步放宽：

- 国家
- 性别偏好，如果产品有配置
- 熟人 / 陌生人
- 胜率 / 技能分
- 历史同桌关系

### 8.4 等待时间设计

推荐目标：

- P50 匹配时间：8 秒以内。
- P90 匹配时间：25 秒以内。
- 单次等待上限：60 秒。
- 用户可随时取消，不扣门票，不进入游戏。

等待阶段：

| 时间 | 策略 | 前台表现 |
| --- | --- | --- |
| 0-5 秒 | 严格匹配同游戏、同语言、同地区、同门票档。 | 展示“正在寻找同桌玩家”。 |
| 5-15 秒 | 优先补入已有等待房；允许匹配同语言但不同国家用户。 | 展示头像占位、已找到人数。 |
| 15-30 秒 | 可创建新房等待补人；允许放宽门票档或人数配置，需用户确认。 | 展示“还差 1 人，可邀请好友”。 |
| 30-45 秒 | 提供选择：继续等待、进入房间先等人、切换热门游戏、邀请好友。 | 弹出轻确认，不强打断。 |
| 45-60 秒 | 兜底：进入等待房或结束匹配。 | 展示“暂时没有合适玩家”。 |

不建议超过 60 秒继续无反馈等待。游戏房是轻娱乐场景，长等待会明显损伤转化。

### 8.5 匹配优先级

优先级从高到低：

1. 先填充已有公开等待房，提升房间活跃度。
2. 再匹配同一批队列用户，创建新房。
3. 仍不足时，引导进入房间等人或邀请好友。
4. 游戏支持机器人且 Bytesun 明确支持时，可在低峰期补机器人；未确认前不作为 V1/V2 必选项。

### 8.6 座位预占

为避免多人同时抢同一个位置：

- 匹配成功后服务端预占座位，TTL 建议 15 秒。
- 用户进入房间并 H5 加载成功后，预占转为正式入座。
- 用户取消、超时、加载失败时释放座位。
- 如果 H5 返回 `type=18` 上座失败，App 释放预占并重新匹配或提示用户。

### 8.7 匹配状态机

状态建议：

1. `idle`：未匹配。
2. `queuing`：排队中。
3. `matched`：找到房间或用户。
4. `seat_reserved`：座位预占。
5. `entering_room`：进入我方游戏房。
6. `loading_game`：打开 Bytesun H5。
7. `game_ready`：收到 `type=20` 或 `gameLoaded`。
8. `seated`：游戏座位同步成功。
9. `playing`：收到 `type=13` 或状态上报 `game_start`。
10. `settling`：结算中。
11. `finished`：完成。
12. `cancelled`：用户取消。
13. `failed`：匹配失败 / 加载失败 / 上座失败。

### 8.8 匹配风控

需要限制：

- 同设备多账号刷局。
- 同 IP 多账号异常同桌。
- 频繁秒退、恶意取消。
- 结算异常、退款异常。
- 黑名单用户参与金币局。
- 低余额用户反复进入付费匹配。
- 未成年或地区合规限制。

建议规则：

- 30 分钟内取消匹配 >= 5 次，进入短时冷却。
- 10 分钟内进入游戏后 30 秒内退出 >= 3 次，降低匹配优先级。
- 同设备 / 同 IP 用户默认不匹配到同一金币局，除非是好友私密房。
- 被举报或封禁用户只能进入观战或不可进入游戏房。

## 11. 技术与产品风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| `hideLobby` 快速开始 API 未在文档给出 | Ludo / UNO 无法直接无感开局 | V0 向 Bytesun 补接口；V1 不依赖；V2 再接入。 |
| 钱包并发扣款 | 余额错账、重复扣款 | `/change_balance` 做用户级锁、订单幂等、流水对账。 |
| H5 加载失败 | 进入房间后白屏或卡 loading | 增加加载超时、重试、降级 URL、错误上报。 |
| 游戏座位和麦位不一致 | 用户看到的玩家关系错乱 | 我方房间座位为准，通过 `gameActionUpdate type=4` 同步。 |
| 画猜聊天违规 | 内容安全风险 | App 聊天审核后再同步游戏。 |
| 游戏声音和房间声音冲突 | 体验混乱 | 提供音乐/音效开关，默认房间语音优先。 |
| MENA 延迟 | 游戏卡顿 | 默认 gsp 选迪拜 201，按地区动态选择。 |
| 三方游戏版本更新不可控 | 线上突发兼容问题 | 游戏包版本灰度、CDN 预热、后台一键下架。 |

## 12. 待向 Bytesun 确认的问题

1. Ludo、UNO、你画我猜对应的准确 `game_id`、游戏名、是否 game_id 以 3 开头。
2. `gamelist` 的 `game_list_type` 到底传 2 还是 3。
3. `hideLobby=true` 对 Ludo / UNO 的快速开始游戏 API 具体路径、参数、鉴权、返回值。
4. Ludo / UNO 是否支持机器人补位，机器人是否会在 `game_start` / `game_settle` 中以 `is_ai=1` 上报。
5. 每个游戏的最小开局人数、最大人数、是否允许中途加入、是否允许观战。
6. 门票档位 `ticketSlots` 的含义、取值和对应金额。
7. `gameMode=[1,2,3]` 中各模式差异；语聊房是否所有游戏都必须使用模式 3。
8. game_id 3 开头游戏的安全区参数是否所有全屏语聊房游戏都必须配置。
9. 你画我猜是否完全依赖 App 聊天同步，还是有内置聊天。
10. 游戏维护、封禁、受限 IP 等错误码是否会透传到 H5，App 是否需要单独展示错误页。

## 13. 推荐结论

1. 先把“游戏房”作为语聊房的新房间类别上线，而不是把用户直接导到三方游戏大厅。
2. V1 先做稳定接入：Ludo、UNO、创建/加入/观战、钱包、结算、基础后台。
3. V1.1 再做你画我猜，因为它需要 App 聊天区和游戏双向同步，内容审核成本更高。
4. 随机匹配建议由我方做，V2 单独成版本，避免 MVP 被复杂匹配拖慢。
5. `hideLobby=true` 是提升 Ludo / UNO 快速开局体验的关键，但必须等 Bytesun 补齐快速开始 API 后再做无感开局。
6. 游戏房的核心体验不是“游戏能打开”，而是“用户能快速找到人、能说话、能开局、能结算、能再来一局”。
