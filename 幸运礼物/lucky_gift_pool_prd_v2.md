# 幸运礼物玩法 & 奖池回收模型落地方案 PRD v2.0

> 适用场景：中东语聊房 / 直播房间 / 公会活动 / 大额送礼氛围玩法  
> 核心目标：在保证平台成本可控的前提下，提供“幸运感、排面、房间氛围、榜单竞争”，避免设计成强博彩感的金币返奖玩法。

---

## 1. 结论先行

### 1.1 推荐方向

不建议把幸运礼物设计成「用户送礼后随机返金币」的模式。更推荐设计为：

```text
幸运礼物 = 正常送礼收益 + 房间奖池进度 + 概率触发特效 + 固定榜单奖励 + 可控奖池返奖
```

其中，奖池返奖如果必须做，需要满足以下原则：

1. **奖池先有钱，再开奖**：大奖不能无资金来源地无限爆。
2. **奖池不足时不爆大奖**：只开放奖池余额可覆盖的奖励档位。
3. **平台可设置补贴浮动范围**：但补贴必须有预算上限、触发条件和熔断机制。
4. **概率不是固定死表**：概率应根据奖池水位、平台预算、用户风控状态动态调整。
5. **奖励尽量虚拟化**：优先发放礼物券、装扮、头像框、贡献值、房间特效，不建议直接返可提现金币。
6. **前端弱化“赢钱感”**：不使用 Jackpot、Bet、Win Cash、翻倍赢、提现等奖池赌博表达。

---

## 2. 幸运礼物玩法分类

| 类型 | 说明 | 是否有奖池 | 是否推荐 | 适配中东程度 | 风险 |
|---|---|---:|---:|---:|---|
| 随机倍数返金币 | 送礼后随机返 1x/2x/10x 金币 | 可无 | 低 | 中 | 高 |
| 房间奖池返奖 | 部分礼物消耗进入奖池，按水位开奖 | 有 | 中 | 中高 | 中高 |
| 房间能量进度 | 礼物转化为能量，满级触发固定奖励 | 可无真实奖池 | 高 | 高 | 低 |
| 暴击特效 | 概率触发大动画/飘屏/横幅 | 无 | 高 | 高 | 低 |
| 榜单奖励 | Top 用户获得固定奖励 | 无 | 高 | 高 | 低 |
| 保底成长 | 送满 N 次得固定奖励 | 无 | 高 | 高 | 中低 |
| 盲盒礼物 | 开盒获得不同礼物/装扮 | 可无 | 中 | 中 | 中 |

### 2.1 中东产品推荐组合

```text
主玩法：房间能量进度 + 榜单奖励 + 暴击特效
可选玩法：低风险奖池返奖
不建议主推：高倍数返金币 / 可提现奖池 / 用户下注式玩法
```

## 3. 推荐产品方案总览

### 3.1 用户流程

```text
用户进入房间
  ↓
选择幸运礼物
  ↓
送出幸运礼物
  ↓
主播获得正常礼物收益
  ↓
用户获得财富值 / 活动贡献值
  ↓
房间幸运能量增加
  ↓
系统判断是否触发幸运事件
  ↓
若触发：展示房间特效 / 飘屏 / 榜单变化 / 奖励弹窗
  ↓
若房间能量达到阶段阈值：触发阶段奖励
  ↓
周期结束后发放榜单奖励
```

### 3.2 核心模块

| 模块 | 作用 |
|---|---|
| 幸运礼物 | 付费入口，用户通过送礼参与玩法 |
| 房间能量 | 替代强博彩奖池的进度表达 |
| 幸运事件 | 概率触发动画、飘屏、横幅、奖励 |
| 贡献榜 | 大R竞争和中东用户排面需求 |
| 阶段奖励 | 房间达到不同等级后发固定奖励 |
| 奖池回收 | 控制平台返奖成本，防止亏损 |
| 风控系统 | 防止小号、主播自刷、套利、退款风险 |

## 4. 概率设置建议

### 4.1 概率分层

| 事件 | 建议概率 | 奖励建议 | 说明 |
|---|---:|---|---|
| 普通反馈 | 100% | 基础动画、财富值、主播收益 | 每次送礼必有反馈 |
| 小幸运 | 10% - 20% | 小动画、房间气泡 | 高频爽感 |
| 中幸运 | 3% - 8% | 横幅、贡献值加成 | 提升参与感 |
| 大幸运 | 0.5% - 2% | 房间大动画、全服飘屏 | 制造传播 |
| 超级幸运 | 0.05% - 0.2% | 稀有装扮、称号、礼物券 | 稀缺感 |

### 4.2 概率配置原则

1. **概率控制的是特效等级，不建议直接控制高价值返金币。**
2. **高价值奖励必须受奖池余额、预算余额、用户风控等级共同限制。**
3. **同一用户短时间连续触发高价值奖励，需要进入冷却。**
4. **新用户、退款用户、异常设备、主播关联账号，降低或禁止触发高档奖励。**

# 5. 奖池回收模型落地方案

本章节是评审重点。

## 5.1 要解决的问题

奖池返奖模型必须回答以下问题：

| 问题 | 必须有明确方案 |
|---|---|
| 用户送礼的钱如何拆分？ | 需要资金流公式 |
| 奖池的钱从哪里来？ | 需要奖池入账规则 |
| 主播收益是否受影响？ | 需要收益口径 |
| 平台怎么保证不亏？ | 需要奖池保护线和预算上限 |
| 奖池不够爆大奖怎么办？ | 需要奖项降级和禁止开奖机制 |
| 是否允许平台垫付？ | 需要补贴池和熔断规则 |
| 概率是否固定？ | 需要动态概率规则 |
| 用户看到什么？ | 需要前端表达和规则页 |
| 后台怎么配置？ | 需要字段表 |
| 财务怎么对账？ | 需要数据口径 |

---

## 5.2 基础定义

### 5.2.1 资金字段定义

| 字段 | 英文 key | 说明 |
|---|---|---|
| 礼物支付金额 | gift_cost | 用户送幸运礼物消耗的金币数 |
| 主播收益部分 | host_income_part | 按平台礼物分成规则计入主播收益 |
| 奖池注入部分 | pool_in_part | 注入幸运奖池的部分 |
| 平台沉淀部分 | platform_margin_part | 平台收入 / 手续费 / 风险准备金 |
| 平台补贴部分 | platform_subsidy_part | 当允许补贴时，平台额外承担的返奖成本 |
| 奖池余额 | pool_balance | 当前可用于返奖的奖池余额 |
| 奖池保护线 | pool_protect_balance | 不允许被消耗的最低奖池余额 |
| 可开奖金额 | pool_available_balance | 可用于派奖的奖池金额 |
| 单次派奖金额 | reward_cost | 本次触发奖励对应成本 |

### 5.2.2 基础公式

```text
礼物支付金额 = 主播收益部分 + 奖池注入部分 + 平台沉淀部分
```

```text
gift_cost = host_income_part + pool_in_part + platform_margin_part
```

```text
可开奖金额 = 奖池余额 - 奖池保护线
```

```text
pool_available_balance = pool_balance - pool_protect_balance
```

只有当：

```text
pool_available_balance >= reward_cost
```

才允许触发该档位奖励。

## 5.3 资金拆分模型

### 5.3.1 推荐拆分方案

假设用户送出 1,000 金币幸运礼物：

| 去向 | 比例示例 | 金额 | 说明 |
|---|---:|---:|---|
| 主播收益 | 50% | 500 | 与普通礼物收益口径一致或略低 |
| 奖池注入 | 25% | 250 | 用于后续返奖 |
| 平台沉淀 | 25% | 250 | 平台收入、风险准备金、运营成本 |

建议初始配置：

```text
主播收益比例：45% - 55%
奖池注入比例：15% - 30%
平台沉淀比例：20% - 40%
```

### 5.3.2 不同礼物价格可配置不同注入比例

| 礼物价格 | 奖池注入比例 | 原因 |
|---:|---:|---|
| 100 金币 | 10% | 低价礼物高频，不宜过度返奖 |
| 500 金币 | 15% | 中低价礼物，适度进入奖池 |
| 1,000 金币 | 20% | 主力礼物，可承担奖池增长 |
| 5,000 金币 | 25% | 大R礼物，提高奖池成长感 |
| 10,000 金币 | 30% | 高价值礼物，适合触发房间氛围 |

### 5.3.3 主播收益是否受影响

建议方案：

| 方案 | 说明 | 是否推荐 |
|---|---|---:|
| 幸运礼物与普通礼物同分成 | 主播收益不变，平台从自己收入中拿部分进奖池 | 高 |
| 幸运礼物主播收益略低 | 主播收益少一部分，转入奖池 | 中 |
| 主播收益完全不计 | 用户只参与抽奖，主播无收益 | 低 |

推荐：**幸运礼物仍然计入主播收益和魅力值**，否则主播没有动力引导用户参与。

## 5.4 奖池类型设计

### 5.4.1 奖池维度

| 奖池类型 | 说明 | 推荐程度 |
|---|---|---:|
| 全平台奖池 | 所有房间共享一个奖池 | 中 |
| 国家奖池 | 按国家/地区独立奖池 | 高 |
| 房间奖池 | 每个房间独立奖池 | 中高 |
| 活动奖池 | 每个活动周期独立奖池 | 高 |
| 用户个人奖池 | 每个用户独立累计 | 低 |

推荐组合：

```text
活动奖池 + 国家奖池 + 房间展示进度
```

也就是：

- 财务真实奖池按活动 / 国家维度统计；
- 前台房间展示的是“幸运能量进度”；
- 不建议每个小房间都做真实独立奖池，容易出现奖池太小无法派奖的问题。

### 5.4.2 为什么不建议纯房间独立奖池

小房间流水低，奖池余额很难覆盖大奖，用户会长期看不到大奖；大房间流水高，大奖频繁，平台成本压力大。若所有房间完全独立，会出现体验不均衡。

推荐做法：

```text
真实资金池：活动 / 国家维度
前台展示池：房间能量 / 房间进度
派奖限制：同时参考真实奖池余额 + 房间贡献 + 用户风控
```

## 5.5 奖项档位设计

### 5.5.1 奖项成本表

| 奖项档位 | 奖励类型 | 奖励成本 reward_cost | 触发条件 |
|---|---|---:|---|
| R1 小奖 | 礼物券 / 小特效 | 50 - 200 金币等值 | 高频触发 |
| R2 中奖 | 头像框碎片 / 贡献加成 | 500 - 1,000 金币等值 | 奖池余额充足 |
| R3 大奖 | 稀有礼物券 / 高级装扮 | 3,000 - 10,000 金币等值 | 需要奖池水位达标 |
| R4 超级奖 | 7日座驾 / 稀有称号 | 20,000 - 50,000 金币等值 | 需要高水位 + 预算允许 |
| R5 爆奖 | 活动大奖 / 全服广播 | 100,000+ 金币等值 | 仅活动期、强限制 |

### 5.5.2 奖励成本口径

奖励成本必须统一，不然财务和产品会对不齐。

| 奖励类型 | 成本计算建议 |
|---|---|
| 金币返还 | 按返还金币面值计成本 |
| 礼物券 | 按可抵扣金币面值计成本 |
| 头像框 / 座驾 | 按内部配置成本计，不按商城售价全额计 |
| 贡献值加成 | 如果不可提现，可按 0 现金成本处理，但要计运营成本 |
| 房间特效 | 可按 0 现金成本处理 |
| 主播魅力值 | 若影响结算，需要计入成本；不影响结算则按 0 处理 |

## 5.6 奖池水位与可开奖档位

### 5.6.1 水位定义

| 水位 | 条件 | 可开放奖项 |
|---|---|---|
| 极低水位 | 可用奖池 < 1,000 | 只开放 R1 |
| 低水位 | 1,000 ≤ 可用奖池 < 10,000 | 开放 R1、R2 |
| 中水位 | 10,000 ≤ 可用奖池 < 50,000 | 开放 R1、R2、R3 |
| 高水位 | 50,000 ≤ 可用奖池 < 200,000 | 开放 R1-R4 |
| 爆奖水位 | 可用奖池 ≥ 200,000 | 开放 R1-R5 |

### 5.6.2 核心规则

```text
系统只在当前奖池水位允许的奖项集合中抽取奖励。
```

例如：

当前可用奖池只有 8,000 金币，则系统只允许抽 R1、R2，不允许抽 R3、R4、R5。

当前可用奖池为 80,000 金币，则系统可以抽 R1-R4，但 R5 不开放。

## 5.7 奖池不足以爆大奖怎么办？

这是奖池模型的关键问题。

### 5.7.1 不允许出现的方案

不建议：

```text
奖池不足，但系统仍然按固定概率爆大奖，差额全部由平台承担。
```

原因：

1. 平台成本不可控；
2. 容易被大R或工作室套利；
3. 活动越火，潜在亏损越大；
4. 财务无法预测周期成本。

### 5.7.2 推荐方案：奖项准入机制

开奖前先判断当前奖池能覆盖哪些奖项。

```text
当前可用奖池 = 当前奖池余额 - 奖池保护线
```

系统根据 `当前可用奖池` 过滤奖项：

```text
可抽奖项 = 奖励成本 <= 当前可用奖池 × 单次最大消耗比例
```

建议配置：

```text
单次最大消耗比例 = 20% - 40%
```

例如：

当前可用奖池 = 50,000 金币  
单次最大消耗比例 = 30%  
则单次最大奖励成本 = 15,000 金币

此时即使奖池有 50,000，也不允许一次性爆 50,000 的大奖，避免奖池被打空。

### 5.7.3 奖池不足时的三种处理

| 处理方式 | 说明 | 推荐程度 |
|---|---|---:|
| 奖项降级 | 原本命中 R4，但奖池不足，降到 R3/R2 | 高 |
| 重新抽取 | 在可用奖项中重新随机 | 高 |
| 转为进度奖励 | 不爆奖，增加房间能量或保底值 | 高 |
| 平台全额垫付 | 奖池不足仍发大奖 | 低 |
| 负奖池 | 奖池余额变成负数，后续流水回补 | 低 |

推荐执行顺序：

```text
命中高奖项
  ↓
判断奖池是否足够
  ↓
足够：正常派奖
  ↓
不足：判断是否允许平台补贴
  ↓
允许且未超预算：平台补贴差额
  ↓
不允许或超预算：奖项降级 / 重新抽取 / 转进度奖励
```

## 5.8 是否需要平台浮动补贴？

建议：**可以有，但必须有限制。**

平台补贴不是无限垫付，而是运营工具，用于保证活动体验。

### 5.8.1 平台补贴池定义

| 字段 | 说明 |
|---|---|
| subsidy_budget_total | 本活动平台总补贴预算 |
| subsidy_budget_used | 已使用补贴金额 |
| subsidy_budget_remaining | 剩余补贴预算 |
| subsidy_single_limit | 单次最大补贴金额 |
| subsidy_daily_limit | 每日最大补贴金额 |
| subsidy_ratio_limit | 单次奖励中平台最多承担比例 |
| subsidy_trigger_waterline | 允许补贴的最低奖池水位 |

### 5.8.2 推荐配置

| 配置 | 建议值 | 说明 |
|---|---:|---|
| 平台总补贴预算 | 活动预期流水的 3% - 8% | 不建议超过 10% |
| 单次最大补贴 | 奖励成本的 20% - 40% | 平台只补差额的一部分 |
| 每日补贴上限 | 总补贴预算的 20% - 30% | 防止一天打爆预算 |
| 触发水位 | 中水位以上 | 低水位不补贴大奖 |
| 高风险用户 | 不允许补贴 | 防套利 |
| 新房间 / 新主播 | 默认不补贴 R4/R5 | 防自刷 |

### 5.8.3 补贴计算公式

```text
奖池可承担金额 = pool_available_balance × single_reward_max_pool_ratio
```

```text
平台可补贴金额 = min(
  reward_cost - 奖池可承担金额,
  subsidy_single_limit,
  subsidy_budget_remaining,
  reward_cost × subsidy_ratio_limit
)
```

只有当：

```text
奖池可承担金额 + 平台可补贴金额 >= reward_cost
```

才允许发放该奖励。

否则必须降级。

### 5.8.4 示例

假设：

```text
R4 大奖成本 = 50,000 金币
当前可用奖池 = 120,000 金币
单次最大奖池消耗比例 = 30%
平台单次补贴上限 = 20,000 金币
平台补贴比例上限 = 30%
平台剩余补贴预算 = 100,000 金币
```

计算：

```text
奖池最多承担 = 120,000 × 30% = 36,000
缺口 = 50,000 - 36,000 = 14,000
平台最多可补 = min(14,000, 20,000, 100,000, 50,000 × 30%) = 14,000
```

结果：

```text
36,000 奖池承担 + 14,000 平台补贴 = 50,000
允许发放 R4
```

如果平台剩余补贴预算只有 5,000：

```text
36,000 + 5,000 = 41,000 < 50,000
不允许发放 R4，降级到 R3
```

## 5.9 动态概率模型

### 5.9.1 为什么不能用固定概率

如果固定概率不看奖池余额，会出现：

1. 奖池低时仍爆大奖，平台亏损；
2. 奖池高时小奖过多，用户觉得没刺激；
3. 活动周期内成本不可控；
4. 风控用户也可能命中高奖。

### 5.9.2 推荐：基础概率 × 调节系数

```text
最终概率 = 基础概率 × 奖池水位系数 × 用户风控系数 × 房间质量系数 × 预算系数
```

### 5.9.3 各系数说明

#### A. 基础概率

| 奖项 | 基础概率 |
|---|---:|
| R1 | 15% |
| R2 | 5% |
| R3 | 1% |
| R4 | 0.2% |
| R5 | 0.02% |

#### B. 奖池水位系数

|---|---:|---:|---:|---:|---:|

| 低 | 1.0 | 0.5 | 0 | 0 | 0 |

| 90% - 100% | 0.1 |
| 已超预算 | 0 |

### 5.9.4 概率归一化

当部分奖项因奖池不足被关闭后，需要重新归一化概率。

例如基础概率：

| 奖项 | 基础概率 |
|---|---:|
| R1 | 15% |
| R2 | 5% |
| R3 | 1% |
| R4 | 0.2% |
| R5 | 0.02% |

当前只开放 R1、R2、R3，则 R4/R5 直接置 0。剩余奖项可按业务选择：

方案一：不归一化，未中奖概率增加。  
方案二：归一化到原奖励总概率，保持中奖率不变但大奖关闭。

推荐：**不做完全归一化，只做小幅补偿。**

原因：奖池低时本来就应该降低整体返奖率，保护平台成本。

## 5.10 奖池保护线

### 5.10.1 为什么需要保护线

如果奖池每次都可以被消耗到 0，会出现：

1. 后续用户体验突然变差；
2. 奖池波动过大；
3. 大奖刚出后房间冷掉；
4. 平台补贴压力增加。

### 5.10.2 保护线配置

| 奖池规模 | 建议保护线 |
|---|---:|
| 小奖池 | 30% - 50% |
| 中奖池 | 20% - 30% |
| 大奖池 | 10% - 20% |
| 活动末期 | 可降低至 5% - 10% |

公式：

```text
pool_protect_balance = pool_balance × protect_ratio
```

```text
pool_available_balance = pool_balance - pool_protect_balance
```

## 5.11 奖池负数是否允许？

### 5.11.1 不推荐默认允许负奖池

负奖池代表平台先垫付，后续靠用户流水回补。风险是：

1. 如果活动后续流水不足，平台真实亏损；
2. 大R可能在活动初期利用高概率窗口套利；
3. 财务对账复杂；
4. 用户投诉时解释难度大。

### 5.11.2 可选：运营白名单负奖池

仅在大型活动中，可以开放轻度负奖池：

| 配置 | 建议 |
|---|---|
| 是否允许负奖池 | 默认否 |
| 最大负奖池额度 | 活动预期流水的 1% - 3% |
| 负奖池开放对象 | 仅官方活动 / 高流水国家 |
| 负奖池期间 | 关闭 R5，降低 R4 概率 |
| 回补方式 | 后续奖池注入优先抵扣负余额 |

如果启用负奖池：

```text
后续每笔奖池注入先用于归还负奖池，直到 pool_balance >= 0
```

## 5.12 开奖流程

### 5.12.1 主流程

```text
用户送出幸运礼物
  ↓
扣除用户金币
  ↓
按比例拆分：主播收益 / 奖池注入 / 平台沉淀
  ↓
更新奖池余额
  ↓
判断用户是否可参与幸运返奖
  ↓
读取当前奖池水位
  ↓
过滤当前可开奖项
  ↓
计算动态概率
  ↓
执行随机命中
  ↓
判断命中奖项是否可支付
  ↓
可支付：扣减奖池/补贴池并发奖
  ↓
不可支付：降级/重抽/转进度奖励
  ↓
写入派奖记录和资金流水
  ↓
前端展示结果
```

### 5.12.2 伪代码

```pseudo
function sendLuckyGift(user, room, gift):
    cost = gift.price

    deductUserCoins(user, cost)

    hostIncome = cost * hostIncomeRatio
    poolIn = cost * poolInRatio
    platformMargin = cost - hostIncome - poolIn

    addHostIncome(room.host, hostIncome)
    addPoolBalance(poolId, poolIn)
    addPlatformMargin(platformMargin)

    if not isUserEligible(user, room):
        return normalGiftResult()

    poolBalance = getPoolBalance(poolId)
    protectBalance = poolBalance * protectRatio
    availableBalance = poolBalance - protectBalance

    rewardCandidates = filterRewardsByAvailableBalance(
        availableBalance,
        singleRewardMaxPoolRatio
    )

    if rewardCandidates is empty:
        return energyOnlyResult()

    finalProbabilities = calculateDynamicProbabilities(
        rewardCandidates,
        poolWaterLevel,
        userRiskLevel,
        roomQuality,
        subsidyBudget
    )

    reward = randomHit(finalProbabilities)

    if reward is null:
        return normalLuckyResult()

    paymentResult = checkRewardPayment(
        reward.cost,
        availableBalance,
        subsidyBudget,
        subsidyRules
    )

    if paymentResult.canPay:
        deductPool(paymentResult.poolPay)
        deductSubsidyBudget(paymentResult.subsidyPay)
        grantReward(user, reward)
        writeRewardLog(user, room, reward, paymentResult)
        return rewardResult(reward)
    else:
        downgradedReward = downgradeReward(reward, rewardCandidates)
        if downgradedReward exists:
            grantReward(user, downgradedReward)
            writeRewardLog(user, room, downgradedReward, paymentResult)
            return rewardResult(downgradedReward)
        else:
            return energyOnlyResult()
```

## 5.13 奖项降级规则

### 5.13.1 降级优先级

当命中奖项无法支付时：

```text
R5 → R4 → R3 → R2 → R1 → 仅展示特效/增加能量
```

### 5.13.2 降级条件

| 条件 | 处理 |
|---|---|
| 奖池余额不足 | 降级到可支付最高档 |
| 补贴预算不足 | 降级到不需要补贴的档位 |
| 用户风控异常 | 降级到 R1 或仅特效 |
| 房间风控异常 | 关闭 R3 以上奖励 |
| 活动预算接近上限 | 关闭 R4/R5 |
| 单用户冷却中 | 只允许 R1/R2 |

### 5.13.3 前端展示

不建议告诉用户“你本来中了 R5，但因为奖池不够降级”。

用户只看到最终奖励结果，例如：

```text
Congratulations! You triggered Lucky Bonus.
```

或：

```text
Lucky Energy +1000
Room Treasure is charging...
```

## 5.14 保底机制

### 5.14.1 为什么需要保底

纯概率会导致部分用户长期无反馈，影响继续参与。保底可以提升体验，但不能设计成“稳赚”。

### 5.14.2 推荐保底

| 保底类型 | 规则 | 奖励 |
|---|---|---|
| 次数保底 | 用户每送 30 次幸运礼物 | 必触发一次小幸运 |
| 消费保底 | 用户累计消费 50,000 金币 | 获得限定称号 / 礼物券 |
| 房间保底 | 房间累计 300,000 能量 | 开启房间宝箱 |
| 活动保底 | 活动期贡献达到门槛 | 发固定榜单奖励 |

### 5.14.3 保底与奖池关系

保底奖励必须预先计入成本。

```text
预计保底成本 <= 活动预算 - 概率返奖预算 - 平台补贴预算
```

如果是礼物券保底，建议设置：

- 有效期 1-3 天；
- 不可提现；
- 不可转赠；
- 不计入二次返奖；
- 可限制只能送指定礼物。

## 5.15 平台成本控制模型

### 5.15.1 活动预算公式

```text
活动最大可承受成本 = 预计活动流水 × 最大返奖成本率
```

例如：

```text
预计活动流水 = 10,000,000 金币
最大返奖成本率 = 12%
最大可承受成本 = 1,200,000 金币
```

这 1,200,000 需要覆盖：

```text
概率返奖成本 + 保底成本 + 平台补贴成本 + 榜单奖励成本
```

### 5.15.2 成本拆分建议

| 成本类型 | 建议占活动流水 |
|---|---:|
| 概率返奖 | 3% - 6% |
| 保底奖励 | 2% - 4% |
| 榜单奖励 | 2% - 5% |
| 平台补贴 | 3% - 8% |
| 总成本 | 8% - 15% |

### 5.15.3 熔断规则

| 触发条件 | 系统动作 |
|---|---|
| 总成本达到预算 70% | 降低 R4/R5 概率 |
| 总成本达到预算 85% | 关闭 R5 |
| 总成本达到预算 95% | 关闭 R4，仅保留 R1-R3 |
| 总成本达到预算 100% | 关闭概率返奖，仅保留固定特效和能量 |
| 单日成本超过日预算 | 当日关闭高奖项 |
| 单房间异常高中奖 | 该房间进入观察 / 降权 |
| 单用户异常高中奖 | 用户进入冷却 / 风控复核 |

## 5.22 验收标准

### 5.22.1 产品验收

| 验收项 | 标准 |
|---|---|
| 用户可送幸运礼物 | 可正常扣费、播放动画、计入贡献 |
| 房间能量正常增长 | 按配置比例增长，无重复计算 |
| 奖励概率生效 | 不同水位下奖励档位符合配置 |
| 奖池不足不爆大奖 | 低水位下 R4/R5 不触发 |
| 奖励可降级 | 不可支付时降级到可支付档位 |
| 平台补贴受限 | 不超过单次/每日/总预算 |
| 风控用户受限 | 黑名单、新号、关联号不能触发高奖 |
| 规则页可查看 | 展示概率、奖励、有效期、限制说明 |

### 5.22.2 技术验收

| 验收项 | 标准 |
|---|---|
| 资金拆分准确 | gift_cost = host_income + pool_in + margin |
| 奖池流水完整 | 每次 pool_in/pool_out 都有记录 |
| 幂等处理 | 重试不重复扣费、不重复派奖 |
| 并发安全 | 多用户同时送礼时奖池余额不被超扣 |
| 配置实时生效 | 后台开关、概率、预算可按配置生效 |
| 熔断可用 | 成本超限后自动关闭高奖项 |
| 数据看板准确 | 流水、成本、奖池余额可对账 |

### 5.22.3 财务验收

| 验收项 | 标准 |
|---|---|
| 每日奖池可对账 | 期初 + 注入 - 支出 = 期末 |
| 补贴支出可追踪 | 每笔补贴有活动、用户、奖励记录 |
| 主播收益口径明确 | 幸运礼物是否计入结算明确 |
| 成本率可控 | 不超过后台配置的最大成本率 |
| 异常记录可追溯 | 降级、失败、补发都有日志 |

## 6. 最终建议

如果目标是中东语聊房长期运营，建议采用以下方案：

```text
日常版本：幸运礼物 + 房间能量 + 暴击特效 + 榜单奖励
活动版本：增加奖池回收模型，但只允许奖池可覆盖的奖励档位开奖
大型活动版本：开放平台有限补贴和高档大奖，但必须有预算、熔断、风控
```

核心原则：

```text
先控成本，再做刺激；先做荣誉，再做返奖；先灰度验证，再开放大奖。
```
