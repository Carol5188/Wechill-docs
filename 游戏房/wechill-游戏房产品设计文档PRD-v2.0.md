## 1. 产品定位

### 1.1 核心定位

游戏房不是单独的游戏App，也不是纯三方游戏大厅，而是"语聊房 + 三方H5游戏"的复合房间类型。

核心目标：

1. 给语聊房增加明确的游戏场景分类，提高房间进入理由。
2. 通过Ludo、UNO、你画我猜等轻游戏增强陌生人破冰、房间留存和互动时长。
3. 复用现有语聊房的用户、麦位、IM、礼物、钱包、风控、房间推荐和运营体系。
4. 通过Bytesun承接游戏玩法、牌局/局内状态、结算逻辑，降低自研游戏成本。

### 1.2 目标用户画像

| 画像 | 特征 | 核心诉求 | 占比预估 |
|------|------|---------|---------|
| 社交型玩家 | 以聊天为主，游戏是润滑剂 | 认识人、找话题 | 40% |
| 休闲玩家 | 想玩游戏但不想下载重度App | 即开即玩、轻量 | 30% |
| 组局党 | 拉朋友一起玩 | 社交裂变、好友互动 | 15% |
| 观战党 | 不想玩但想看 | 内容消费、弹幕互动 | 10% |
| 竞技党 | 想赢、想上分 | 排行榜、段位 | 5% |

### 1.3 与现有功能的关系

| 功能 | 游戏房接入点 | 说明 |
|------|-------------|------|
| 红包返奖（Lucky Pocket） | 游戏结算后可触发返奖抽奖 | 复用返奖分档和风控体系 |
| 龙蛋玩法 | 游戏贡献值计入龙蛋进度 | 按分区独立配置，GMT+3统计 |
| CP/Soul Pair | 游戏内亲密值来源（一起玩） | 同游戏可获得亲密值加成 |
| 风控体系 | 复用红包风控黑名单机制 | 危险级用户禁止进入游戏房 |

## 2. Bytesun对接关键约束

> 本节所有内容来自 BytesunGame（语聊房模式）对接文档1.0.7，作为产品设计的前置约束。

### 2.1 接入方式

- 商户侧调用Bytesun游戏信息列表接口，拿到游戏Zip包下载地址或URL地址、游戏版本号。
- 客户端按版本号处理游戏包更新，Zip包可下载并存储到App本地。
- 用户访问解压后的本地路径或URL，进入游戏H5界面。
- 游戏内下注、结算时，Bytesun服务端调用我方服务端API修改用户游戏币。
- 游戏结束后回到我方App/语聊房。
- Bytesun游戏离线Zip包默认是OSS源站；为体验建议我方配置CDN回源，缓存策略建议大于3天，并做预热。

### 2.2 前端JSBridge

客户端需要支持Bytesun H5调用App，也需要App主动调用游戏H5。

**H5调App：**

| 方法 | 说明 | 语聊房模式注意 |
|------|------|---------------|
| `getConfig` | 获取商户、用户、房间、语言、角色、节点等配置 | 必须实现 |
| `destroy` | 非语聊房模式下游戏主动关闭WebView | 语聊房模式通常不依赖 |
| `gameRecharge` | 余额不足或点击金币图标时，通知App打开商城 | 必须实现 |
| `gameLoaded` | 游戏加载完毕，App可关闭自己的加载动画 | 必须实现 |
| `sendGameAction` | 游戏主动通知App（上下座、踢人、开始/结束等） | 必须实现，详见2.2.1 |

**App调H5：**

| 方法 | 说明 |
|------|------|
| `walletUpdate` | App充值或货币变化后通知游戏刷新余额 |
| `gameActionUpdate` | App通知游戏座位操作、身份变更、踢人结果、聊天同步、最小化状态 |
| `soundUpdate` | 设置游戏背景音乐和音效开关 |

#### 2.2.1 sendGameAction 完整type清单

| type | 场景 | 参数 | 产品注意 |
|------|------|------|---------|
| 7 | 拉起用户资料卡 | userId | 点击游戏内用户头像触发 |
| 13 | 游戏开始 | - | App记录开局时间、更新房间状态 |
| 14 | 游戏结束 | - | App展示结算、触发返奖判断 |
| 15 | 上/下座位 | optType(0上/1下), userId, seat | App需鉴权后通过type=4返回 |
| 16 | 座位信息同步 | seats[] | 以App麦位为准，不一致时主动同步 |
| 17 | 发起踢人 | userId, seat | App需鉴权后通过type=6返回 |
| 18 | 上座失败 | userId | App释放预占、重新匹配或提示 |
| 20 | 语聊房准备完成 | - | App可执行后续操作（自动上座等） |
| 21 | 查询音乐/音效状态返回 | bgmStatus, seStatus | 收到type=2012时发起 |
| 22 | 画猜消息同步到App | isSystem, isPlayer, userId, nickName, pos, content, headImage | 需内容审核后再展示 |
| 23 | 上报游戏参数 | isMall等 | 游戏加载完成后上报 |
| 30 | 最大人数/门票变更 | platRoomID, peopleNum, ticketSlots | 动态调整房间配置 |
| 3001 | RTC麦克风/扬声器同步 | status, roomId, users[] | 狼人杀/谁是卧底专用 |

#### 2.2.2 gameActionUpdate 完整type清单

| type | 场景 | 参数 | 产品注意 |
|------|------|------|---------|
| 4 | 操作游戏座位 | optType(0上/1下), seat, userId | 以我方房间座位为准 |
| 5 | 变更用户身份 | role, userId | 观众转玩家、玩家转主持人 |
| 6 | 返回踢人结果 | optResult(0成功/1失败), optUserId, userId, reason | 踢人鉴权结果 |
| 2012 | 查询音乐/音效状态 | - | 通过type=21返回结果 |
| 2014 | 聊天内容同步到游戏 | userId, content | 仅同步审核通过的内容 |
| 2016 | 最小化状态变更 | collapseStatus(0开始收起/1收起完成/2开始展开/3展开完成) | 缩小球状态控制 |

### 2.3 getConfig关键字段口径

| 字段 | 类型 | 设计口径 |
|------|------|---------|
| `appChannel` | string | Bytesun提供，后台配置 |
| `appId` | int64 | Bytesun提供的商户ID |
| `userId` | string | 我方用户ID |
| `code` | string | 我方生成的一次性认证令牌。**必须唯一且一次性**，Bytesun用该参数向我方服务端换`ss_token`，换完后该code不可再用，再次使用我方返回1001错误码 |
| `roomId` | string | 我方语聊房/游戏房房间ID |
| `gameRoomId` | string | 文档要求传空字符串，保留字段 |
| `gameMode` | string | 语聊房场景**固定传"3"** |
| `language` | string | 语言值：英文`2`，阿语`7`，中文`0` |
| `gameConfig.sceneMode` | int | 场馆级别，默认0 |
| `gameConfig.currencyIcon` | string | 货币图标，需外网可访问URL，60×60 |
| `gsp` | int | 游戏服务器节点：**MENA优先201(迪拜AWS)**，必要时降级101(新加坡) |
| `role` | int | 0=正常用户可游戏可观战，1=游客仅观战，2=主持人可游戏可观战可开启游戏 |

### 2.4 服务端必须提供的API

| 接口 | 用途 | 关键要求 |
|------|------|---------|
| `/v1/api/get_sstoken` | Bytesun用一次性code换长期ss_token | code必须一次性消费，再次使用返回1001 |
| `/v1/api/get_user_info` | Bytesun查询用户昵称、头像、余额、类型 | 返回balance，可扩展多币种balance_list；user_type映射我方风控等级 |
| `/v1/api/update_sstoken` | ss_token过期时刷新 | 如果我方ss_token非长期有效，必须实现 |
| `/v1/api/change_balance` | 下注、结算、退款时修改用户货币 | **必须做单用户并发锁**；order_id唯一；余额不足错误码1008；**成功code才能为0** |
| 游戏状态上报接口 | Bytesun上报game_start、game_settle | 用于局记录、数据看板、风控和用户战绩 |

**安全要求：**
- 请求双方用`signature_nonce + AppKey + timestamp`做MD5签名
- signature 15秒内有效
- signature_nonce 15秒内不能重复，防重放
- AppKey必须严格保密

### 2.5 Bytesun游戏信息接口

Bytesun提供：
- `/v1/api/one_game_info`：获取单个游戏信息
- `/v1/api/gamelist`：获取游戏信息列表

返回字段：game_id, name, preview_url, game_version, download_url, game_mode, game_orientation(1竖屏/2横屏), safe_height, venue_level

⚠️ **待确认**：文档字段说明中`game_list_type`写"2游戏"，示例请求写的是`3`，上线前需和Bytesun确认真实取值。

### 2.6 特殊游戏能力

| 能力 | 适用游戏 | 说明 |
|------|---------|------|
| `hideLobby=true` | DominoPlus, ludoPlus, unoPlus | 隐藏大厅，游戏启动若非重连则停留在loading页，**等待我方服务端调用"快速开始游戏API"直接开局**。该API在1.0.7文档中未给出定义，需向Bytesun补充确认 |
| 画猜聊天同步 | 你画我猜 | H5通过type=22发消息到App聊天区；App通过type=2014同步聊天内容回游戏 |
| 游戏内RTC | 狼人杀、谁是卧底 | URL加`isGameRTC=true`，Android需授权音频捕获；**普通Ludo/UNO建议继续使用我方语聊房RTC** |
| 安全区参数 | game_id以3开头的全屏语聊房游戏 | URL加`game_margin_top`、`game_margin_bottom`、`game_margin_standard`，避免游戏操作区被App麦位/顶栏/底栏遮挡 |

## 5. 核心用户链路

### 5.1 创建游戏房

```
1. 房主进入创建页
2. 选择游戏、语言、人数、门票、是否公开
3. 我方创建game_room类型语聊房，生成voice_room_id
4. 服务端生成一次性code
5. 客户端打开Bytesun H5
6. H5调getConfig，App返回：
   - roomId = voice_room_id
   - gameMode = "3"
   - role = 2（房主=主持人）
   - gsp = 201（迪拜）
   - language = "7"（阿语）
7. 收到gameLoaded后关闭App loading动画
8. 收到type=20语聊房游戏准备完成后，App可按需自动同步房主上座
9. 房主邀请用户或等待随机匹配补人
```

### 5.2 加入游戏房

```
1. 用户从游戏Tab、房间推荐、好友邀请、随机匹配进入房间
2. 我方判断房间状态：
   - 等待中/准备中 → 可作为玩家加入
   - 游戏中 → 默认以观众身份加入（除非游戏支持中途加入）
3. 客户端打开对应游戏H5
4. H5调getConfig，App返回用户role：
   - 可参与 → role=0
   - 仅观战 → role=1
5. H5同步座位信息type=16
6. App检查游戏座位和我方麦位/玩家位是否一致
7. 不一致时，App通过gameActionUpdate type=4以我方房间座位为准同步给游戏
```

### 5.3 开局与结算

```
1. 游戏通知App type=13：游戏开始
2. Bytesun服务端上报game_start
3. 游戏内下注时，Bytesun调我方/change_balance扣币（diff_msg="bet"）
4. 余额不足时：
   a. 游戏H5调gameRecharge
   b. App打开商城
   c. 用户充值成功后，App调walletUpdate通知游戏刷新余额
5. 游戏结束时，H5通知App type=14
6. Bytesun服务端上报game_settle
7. 结算奖励时，Bytesun调我方/change_balance加币（diff_msg="result"）或退款（diff_msg="refund"）
8. App展示局后结果：排名、奖励、再来一局、邀请好友、返回房间
```

### 5.4 你画我猜聊天链路（V1.1）

```
1. 游戏需要把猜测消息发到App聊天区时，H5调sendGameAction type=22
2. App根据isSystem、isPlayer、userId、nickName、pos、content展示聊天消息
3. App用户在聊天区输入猜测时，App调gameActionUpdate type=2014把userId和content同步给游戏
4. ⚠️ 我方必须对聊天内容做敏感词、辱骂、广告、涉政涉黄等审核
5. 仅同步审核通过的内容给游戏
6. 违规内容拦截并记录违规拦截量指标
```

### 5.5 游戏声音与房间语音冲突处理

```
场景：游戏中音效 vs 语聊房语音可能冲突

策略：
1. 游戏开始(type=13)时，App调soundUpdate关闭游戏BGM，保留音效
2. 语聊房语音始终优先
3. 用户可手动开关游戏BGM/音效（通过type=2012查询/设置）
4. 游戏结束(type=14)后恢复用户之前的BGM偏好
5. 狼人杀等isGameRTC=true的游戏，type=3001同步麦克风状态：
   - status=0 本局开始加入房间
   - status=1 变更麦克风/扬声器状态
   - status=2 本局结束退出房间
```

### 5.6 最小化/缩球操作

```
场景：用户想缩小游戏画面，回到语聊房主界面

流程：
1. App调用gameActionUpdate type=2016
   - collapseStatus=0：开始收起
   - collapseStatus=1：收起完成
   - collapseStatus=2：开始展开
   - collapseStatus=3：展开完成
2. 最少支持0和3两个状态（收起完成、展开完成）
3. 收起后游戏以缩球形式悬浮展示
4. 用户点击缩球展开回游戏
```

## 7. 随机匹配系统

### 7.1 匹配归属：为什么我方做

结论：**随机匹配由我方实现**，从V2做成平台能力。

原因：

1. 游戏房是我方语聊房的一种房间类别，用户先进入的是我方社交场，不只是Bytesun游戏大厅。
2. 我方需要控制推荐、分区、语言、房间状态、麦位、风控、钱包、礼物、CP、活动任务和数据归因。
3. Bytesun文档只负责游戏H5、局内座位、开局、结算、余额变更，不覆盖我方房间推荐和跨房匹配。
4. `hideLobby=true`虽支持Ludo/UNO，但文档明确说要由App端服务器调用"快速开始游戏API"直接开局；该API在1.0.7文档中未给出定义，需向Bytesun补充确认。
5. 如果完全依赖三方大厅，产品会失去"游戏房类别"的运营能力，也难以保证用户等待体验和回流链路。

**边界划分：**

| 职责方 | 负责内容 |
|-------|---------|
| 我方 | 随机匹配、等待队列、房间创建/路由、语言/地区/风险过滤、座位预占、超时降级、邀请好友、数据看板 |
| Bytesun | 游戏H5、游戏规则、局内状态、牌局/棋局逻辑、结算结果、局内座位事件 |
| 协作 | Ludo/UNO的hideLobby快速开局，需Bytesun提供明确server API；API未确认前V1不依赖 |

### 7.2 匹配入口

| 入口 | 位置 | 版本 |
|------|------|------|
| 随机匹配 | 游戏Tab顶部 | V2 |
| 快速开始 | 游戏卡片 | V2 |
| 缺人中，加入本局 | 房间详情 | V2 |
| 再来一局 | 结算页 | V2 |
| 一起玩 | 好友邀请卡 | V2 |

### 7.3 匹配维度

**第一优先级（必须一致）：**

- game_id
- 语言
- 地区/gsp节点
- 房间状态：等待中或准备中
- 用户身份：非封禁、非高风险
- App版本/游戏包版本可用

**第二优先级（建议一致）：**

- 门票档位/底注
- 最大人数
- 是否允许观战
- 是否允许陌生人
- 用户等级或新老用户层级

**第三优先级（可逐步放宽）：**

- 国家
- 性别偏好（如果产品有配置）
- 熟人/陌生人
- 胜率/技能分
- 历史同桌关系

### 7.4 等待时间设计

推荐目标：
- P50匹配时间：8秒以内
- P90匹配时间：25秒以内
- 单次等待上限：60秒
- 用户可随时取消，不扣门票，不进入游戏

| 时间 | 策略 | 前台表现 |
|------|------|---------|
| 0-5秒 | 严格匹配同游戏、同语言、同地区、同门票档 | 展示"正在寻找同桌玩家" |
| 5-15秒 | 优先补入已有等待房；允许匹配同语言但不同国家用户 | 展示头像占位、已找到人数 |
| 15-30秒 | 可创建新房等待补人；允许放宽门票档或人数配置 | 展示"还差1人，可邀请好友" |
| 30-45秒 | 提供选择：继续等待、进入房间先等人、切换热门游戏、邀请好友 | 弹出轻确认，不强打断 |
| 45-60秒 | 兜底：进入等待房或结束匹配 | 展示"暂时没有合适玩家" |

⚠️ 不建议超过60秒继续无反馈等待。游戏房是轻娱乐场景，长等待会明显损伤转化。

### 7.5 匹配优先级

1. 先填充已有公开等待房，提升房间活跃度
2. 再匹配同一批队列用户，创建新房
3. 仍不足时，引导进入房间等人或邀请好友
4. 游戏支持机器人且Bytesun明确支持时，可在低峰期补机器人；**未确认前不作为V1/V2必选项**

### 7.6 座位预占

为避免多人同时抢同一个位置：

- 匹配成功后服务端预占座位，**TTL=15秒**
- 用户进入房间并H5加载成功后，预占转为正式入座
- 用户取消、超时、加载失败时释放座位
- 如果H5返回type=18上座失败，App释放预占并重新匹配或提示用户

### 7.7 匹配状态机

```
idle → queuing → matched → seat_reserved → entering_room → loading_game → game_ready → seated → playing → settling → finished
  ↓        ↓         ↓          ↓               ↓               ↓            ↓
cancelled  cancelled  failed    failed          failed          failed       failed
```

| 状态 | 说明 |
|------|------|
| idle | 未匹配 |
| queuing | 排队中 |
| matched | 找到房间或用户 |
| seat_reserved | 座位预占（TTL=15s） |
| entering_room | 进入我方游戏房 |
| loading_game | 打开Bytesun H5 |
| game_ready | 收到type=20或gameLoaded |
| seated | 游戏座位同步成功 |
| playing | 收到type=13或game_start上报 |
| settling | 结算中 |
| finished | 完成 |
| cancelled | 用户取消 |
| failed | 匹配失败/加载失败/上座失败 |

### 7.8 匹配风控

| 规则 | 阈值 | 处罚 |
|------|------|------|
| 频繁取消匹配 | 30分钟内取消≥5次 | 短时冷却（10分钟禁止匹配） |
| 秒退 | 10分钟内进游戏后30秒内退出≥3次 | 降低匹配优先级 |
| 同设备/同IP同桌 | 默认检测 | 不匹配到同一金币局（好友私密房除外） |
| 黑名单用户 | 危险级 | 只能观战，不能进入游戏房 |
| 低余额反复匹配付费局 | 余额<门票 | 提示充值，不进入匹配 |

| 预算熔断 | 日返奖预算未熔断（日上限200万金币） |

| 档位 | 门槛 | 概率 | 比例 |
|------|------|------|------|
| A档 | 门票200-499 | 3% | 5% |
| B档 | 门票≥500 | 5% | 10% |

#### 流程

```
游戏结束 → 判断是否满足返奖门槛 → 创建"游戏房间红包" → 发送方收到返奖弹窗 → 记录返奖流水
```

## 11. 技术与产品风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| hideLobby快速开始API未在文档给出 | Ludo/UNO无法直接无感开局 | V0向Bytesun补接口；V1不依赖；V2再接入 |
| 钱包并发扣款 | 余额错账、重复扣款 | /change_balance做用户级锁、订单幂等、流水对账 |

| MENA延迟 | 游戏卡顿 | 默认gsp选迪拜201，按地区动态选择 |
| 三方游戏版本更新不可控 | 线上突发兼容问题 | 游戏包版本灰度、CDN预热、后台一键下架 |
| game_list_type取值歧义 | 游戏列表获取失败 | V0确认真实取值 |
| 结算失败 | 用户金币丢失 | 退款策略+结算失败补偿+流水对账 |

## 15. 推荐结论

1. **先把"游戏房"作为语聊房的新房间类别上线**，而不是把用户直接导到三方游戏大厅。
2. **V1先做稳定接入**：Ludo、UNO、创建/加入/观战、钱包、结算、基础后台。
3. **V1.1再做你画我猜**，因为它需要App聊天区和游戏双向同步，内容审核成本更高。
4. **随机匹配建议由我方做**，V2单独成版本，避免MVP被复杂匹配拖慢。
5. **hideLobby=true是提升Ludo/UNO快速开局体验的关键**，但必须等Bytesun补齐快速开始API后再做无感开局。
6. **游戏房的核心体验不是"游戏能打开"**，而是"用户能快速找到人、能说话、能开局、能结算、能再来一局"。
