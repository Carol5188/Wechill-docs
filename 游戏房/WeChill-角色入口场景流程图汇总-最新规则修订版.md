# WeChill 三方休闲游戏接入 - 入口×场景 流程图汇总（修订版）

> 配套文档：PRD v1.6 可评审版 + v1.7 补充修订  
> 用途：基于正确的角色模型，补充从各入口进入游戏的完整路径、最小化、退出场景流程图  
> 日期：2026-05-21（修订 v2 — 调整 1.3/2.2/5.3/6.2/6.3/6.4 流程）

> 本次修订重点：推荐游戏 15s 匹配、Private room 等待与邀请链路、Discover 进入别人游戏房后的最小化承接、房主退出/关闭游戏后的系统接管、踢人不影响游戏中玩家。

---

## 核心概念说明

### 角色定义

| 角色 | 定义 | 说明 |
|---|---|---|
| 房主 | 用户在自己的房间内 = 房主 | 每个用户都有自己的房间，进入自己房间就是房主；房主离房后房间不立即关闭，其他用户可继续聊天/送礼 |
| 普通用户 | 进入别人房间的人 | 无论什么身份，进入别人的游戏房就是普通用户 |

**核心原则：**
- 每个用户都可以是房主（在自己房间时）
- 游戏房 = 某个房主在自己的房间里开启了游戏
- 其他人进入这个游戏房 = 普通用户
- 不存在独立的"管理员"游戏角色

### 入口说明

| 入口 | 说明 | 用户身份 |
|---|---|---|
| Private room | 进入自己的房间 → 开启游戏房模式 | 房主（自己开局） |
| Discover 推荐 / Game Rooms | 去别人的游戏房 → 加入游戏 | 普通用户（加入别人的局） |
| 房内 Games 面板 | 已经在自己房间内 → 开启游戏 | 房主（自己开局） |
| 在别人房间内 | 已在别人房间 → 加入该房间游戏 | 普通用户 |

## 一、Discover 页面入口流程

### 1.1 Discover 页面整体判断逻辑

> 核心：第一层判断用户当前是否在某个房间（最小化状态）

```mermaid

flowchart TD
    A["用户在 Discover 页面点击游戏/Game Room"] --> B{"第一层判断：用户当前是否在某个房间？"}
    
    B -->|"不在任何房间"| C["直接进入匹配流程"]
    B -->|"在某个房间（已最小化）"| D{"第二层判断：是自己的房间还是别人的房间？"}
    
    D -->|"在自己的房间"| E["提示：需退出当前房间"]
    D -->|"在别人的房间"| F["提示：需退出当前房间"]
    
    E --> G{"用户选择"}
    F --> G
    G -->|"确认退出"| H["退出当前房间（释放麦位/游戏座位等）"]
    G -->|"取消"| I["留在 Discover，不做操作"]
    
    H --> C
    
    C --> J{"用户点击的是什么"}
    J -->|"点击某个 Game Room"| K["以普通用户身份进入该游戏房"]
    J -->|"点击推荐游戏"| L["匹配有该游戏的 Game Room 列表"]
    L --> M["选择一个 Game Room 进入"]
    M --> K
```

### 1.2 进入别人的游戏房（普通用户身份）

```mermaid
flowchart TD
    A["以普通用户身份进入游戏房"] --> B["进入该房主的语聊房"]
    B --> C{"房间权限校验"}
    C -->|"加锁/私密/满员"| D["提示对应错误，无法进入"]
    C -->|"通过"| E["进入房间，展示游戏入口"]
    
    E --> F{"当前游戏状态"}
    F -->|"waiting（等待中）"| G["展示 Join 按钮"]
    F -->|"playing（进行中）"| H["展示 Watch（观战）"]
    F -->|"settling（结算中）"| I["展示 Settling"]
    F -->|"ended（已结束）"| J["展示结束态或新一局入口"]
    
    G --> K["点击 Join"]
    K --> L{"座位是否有空"}
    L -->|"满"| M["提示座位已满，可观战"]
    L -->|"有空"| N{"游戏币 >= 门票"}
    N -->|"不足"| O["弹窗：余额不足 → 跳转兑换页"]
    O --> P["兑换成功 → walletUpdate → 返回"]
    P --> N
    N -->|"充足"| Q["预扣门票（冻结）→ 加入游戏座位"]
    Q --> R["进入 H5 游戏界面，身份 = 普通用户"]
    
    H --> S["点击 Watch → 进入 H5 观战态"]
```

### 1.3 Discover 推荐游戏 → 匹配 Game Room

> 调整后口径：点击推荐游戏先展示 Matching 15s 倒计时；若存在同类型、未开局、等待中的游戏房，直接进入该房间；若没有等待中的游戏房，才走原空结果链路。

```mermaid
flowchart TD
    A["用户在 Discover 点击推荐游戏卡片"] --> B{"用户当前是否在房间？"}
    B -->|"不在房间"| C["展示匹配覆盖层：Matching 15s 倒计时"]
    B -->|"在自己的房间（已最小化）"| D["提示：需先退出当前自己的房间"]
    B -->|"在别人的房间（已最小化）"| E["提示：需先退出当前所在房间"]
    D --> F{"用户选择"}
    E --> F
    F -->|"确认退出"| G["退出当前房间，清理语聊房最小化状态"]
    F -->|"取消"| H["留在 Discover，不发起匹配"]
    G --> C

    C --> I{"15s 内是否发现等待中的同类型游戏房？"}
    I -->|"有：同类型 + waiting/ready + 未开局"| J["直接进入该游戏房"]
    J --> K["以普通用户身份进入房主语聊房"]
    K --> L["展示 Join / Watch，后续同 1.2"]

    I -->|"无：倒计时结束仍无等待房"| M["展示空结果：当前暂无可加入的该游戏房"]
    M --> N{"用户选择"}
    N -->|"自己开一局（Private room）"| O["进入 Private room 流程：回自己房间并创建游戏"]
    N -->|"换个游戏"| P["回到 Discover 游戏列表"]
    N -->|"继续等待/重试"| C
    C -->|"用户点击 Cancel"| Q["取消匹配，回到 Discover"]
```

**匹配规则：**
- 只匹配同一个 `game_id` / `game_key` 的游戏房。
- 只匹配 `waiting/ready` 且未正式开局的房间。
- 多个房间同时满足时，优先进入在线人数接近开局要求、延迟低、风险状态正常的房间。
- 推荐游戏入口进入别人房间后，用户身份始终为普通用户。

## 二、Private Room 入口流程

### 2.1 Private Room = 进入自己房间 + 开启游戏

> Private room 的本质：用户进入自己的房间，以房主身份直接开启游戏房模式

```mermaid
flowchart TD
    A["用户点击 Private room 按钮"] --> B{"第一层判断：用户当前是否在某个房间？"}
    
    B -->|"不在任何房间"| C["进入自己的房间"]
    B -->|"在自己的房间（已最小化）"| D["恢复自己的房间"]
    B -->|"在别人的房间（已最小化）"| E["提示：需退出当前房间才能回到自己的房间"]
    
    E --> F{"用户选择"}
    F -->|"确认退出"| G["退出别人的房间"]
    F -->|"取消"| H["留在当前状态"]
    G --> C
    
    C --> I["展示游戏选择面板（半屏/全屏）"]
    D --> I
    I --> J["用户选择一个游戏"]
    J --> K{"当前自己房间是否已有游戏进行中"}
    K -->|"有"| L["提示：当前已有游戏进行中"]
    K -->|"无"| M["打开游戏设置页"]
    
    M --> N["选择人数 / 模式 / 门票"]
    N --> O["点击 Start"]
    O --> P{"游戏币余额 >= 门票"}
    P -->|"不足"| Q["弹窗 → 兑换页"]
    Q --> R["兑换成功 → 返回"]
    R --> P
    P -->|"充足"| S["预扣门票（冻结）"]
    S --> T["创建 RoomGameSession"]
    T --> U{"创建成功"}
    U -->|"失败"| V["退还冻结门票 → 提示"]
    U -->|"成功"| W["加载 H5 游戏"]
    W --> X["H5 调 getConfig → App 返回配置（role=2 房主）"]
    X --> Y{"gameLoaded（30s 超时）"}
    Y -->|"失败/超时"| Z["退还门票 → Retry / Close"]
    Y -->|"成功"| AA["ticket_order 确认扣费"]
    AA --> AB["展示全屏游戏界面，等待其他人加入"]
```

### 2.2 Private Room 开启后等待其他用户

> 调整后口径：Private room 开启后先展示游戏等待状态。房主可以最小化等待页，也可以分享邀请链接。若等待页已最小化，且有人进入该房间并点击加入游戏，则房主侧弹起游戏配置页面，由房主决定是否开始；人数未达成时继续等待，过程中房主可以调整配置。

```mermaid
flowchart TD
    A["房主通过 Private room 创建 RoomGameSession"] --> B["展示游戏等待状态页面（waiting）"]
    B --> C{"房主操作"}

    C -->|"保持在等待页"| D["等待其他用户加入"]
    C -->|"最小化等待页面"| E["收起 H5/等待页 → 回到语聊房界面"]
    E --> F["展示游戏最小化入口：Waiting / Open"]

    C -->|"分享邀请链接"| G["生成邀请链接/房间卡片"]
    G --> H["用户点击链接进入该房间"]
    H --> I["以普通用户身份进入游戏房"]
    I --> J["点击 Join 加入游戏"]

    D --> K{"是否有用户点击 Join？"}
    F --> K
    J --> K
    K -->|"否"| L["继续等待；游戏房展示在 Discover Game Rooms"]
    L --> D

    K -->|"是"| M["房主视角弹起游戏配置页面"]
    M --> N["房主可调整：人数 / 模式 / 门票等配置"]
    N --> O{"人数是否达到开局要求？"}
    O -->|"未达成"| P["Start 置灰或提示人数不足"]
    P --> Q["继续等待更多用户加入"]
    Q --> M
    O -->|"已达成"| R["房主点击 Start"]
    R --> S["二次校验游戏币/座位/房间状态"]
    S --> T{"校验是否通过？"}
    T -->|"否"| U["提示原因：余额不足/座位变化/房间状态异常"]
    T -->|"是"| V["sendGameAction type=13 开局"]
    V --> W["游戏状态变为 playing，所有已加入用户进入本局"]
```

**补充规则：**
- 房主最小化等待页后，游戏房仍处于 `waiting`，可继续被 Discover / 邀请链接发现。
- 房主弹起配置页不是重新创建游戏，而是对当前 waiting session 做开局前配置确认。
- 配置变更仅在开局前生效；playing 后不允许调整本局配置。
- 若房主长时间不响应，可保留 waiting，但需有超时清理策略，避免列表中长期挂空房。

## 三、房内 Games 面板流程（房主在自己房间）

### 3.1 房主在自己房间内开启游戏

> 用户已经在自己的房间里（是房主），通过 Games 面板开启游戏

```mermaid
flowchart TD
    A["房主在自己房间内"] --> B["点击底部 Games 按钮"]
    B --> C["打开房内半屏面板（游戏列表）"]
    C --> D["选择一个游戏"]
    D --> E{"当前房间是否已有游戏 session"}
    E -->|"有"| F["提示：当前已有游戏进行中"]
    E -->|"无"| G["打开游戏设置页"]
    
    G --> H["选择人数 / 模式 / 门票"]
    H --> I["点击 Start"]
    I --> J{"游戏币余额 >= 门票"}
    J -->|"不足"| K["弹窗 → 兑换页"]
    J -->|"充足"| L["预扣门票 → 创建 RoomGameSession"]
    L --> M{"创建成功"}
    M -->|"失败"| N["退还门票 → 提示"]
    M -->|"成功"| O["加载 H5（role=2）"]
    O --> P{"gameLoaded"}
    P -->|"成功"| Q["确认扣费 → waiting 状态"]
    P -->|"失败/超时"| R["退还门票 → Retry / Close"]
```

**说明：** 房内 Games 面板流程和 Private room 流程的本质一样——都是房主在自己房间开启游戏。区别仅在于入口位置不同：
- Private room：从 Discover 页面进入自己房间再开游戏
- Games 面板：已经在自己房间内，直接开游戏

## 五、最小化场景流程图

```mermaid
flowchart TD
    A["房主点击最小化"] --> B["收起 H5 游戏容器"]
    B --> C["回到自己房间的语聊房界面"]
    C --> D["展示游戏最小化入口（浮窗）"]
    
    D --> E{"当前游戏状态"}
    E -->|"waiting"| F["浮窗：Waiting / Open"]
    E -->|"playing"| G["浮窗：Playing / Open"]
    E -->|"settling"| H["浮窗：Settling"]
    E -->|"ended"| I["浮窗：Finished"]
    
    F --> J["房主点击 Open"]
    G --> J
    J --> K["恢复流程：发送 type=2016 expand → 查询最新状态"]
    K --> L{"session 是否有效"}
    L -->|"有效"| M["恢复 H5，房主保持 Start/Close/Kick 权限"]
    L -->|"ended"| N["展示结算结果"]

```

**房主最小化期间的规则：**
- 游戏继续运行，房主仍是 game_controller
- 普通用户仍可继续 Join / Watch
- 房主恢复后保持完整管理权限

### 5.2 普通用户最小化游戏

```mermaid
flowchart TD
    A["普通用户点击最小化"] --> B["收起 H5"]
    B --> C["回到所在的语聊房界面"]
    C --> D["展示最小化浮窗"]
    
    D --> E{"当前游戏状态"}
    E -->|"waiting"| F["浮窗：Waiting"]
    E -->|"playing"| G["浮窗：Playing"]
    E -->|"settling"| H["浮窗：Settling"]
    E -->|"ended"| I["浮窗：Finished"]
    
    G --> J["点击 Open"]
    J --> K["恢复流程"]
    K --> L{"恢复结果"}
    L -->|"session playing，座位保留"| M["恢复游戏继续玩"]
    L -->|"session playing，但座位被释放"| N["展示 Watch / 重新 Join"]
    L -->|"session ended"| O["展示结算结果"]
    L -->|"被踢出房间"| P["无法恢复，清理入口"]
```

**普通用户最小化期间的风险：**
- 回合制游戏可能超时判负
- 游戏可能结束

### 5.3 从 Discover 进入的用户最小化后的处理

```mermaid
flowchart TD
    A["用户从 Discover 进入别人游戏房"] --> B["游戏中点击最小化"]
    B --> C["收起 H5 → 回到该房间语聊界面"]
    C --> D["展示房内游戏最小化入口：Playing / Open"]
    D --> E{"用户操作"}

    E -->|"点击游戏入口"| F["恢复 H5 游戏界面，继续游戏/观战"]
    E -->|"继续留在语聊房"| G["可聊天、送礼、听麦；游戏按三方规则继续"]
    E -->|"再次最小化房间"| H["走原有语聊房最小化逻辑"]
    H --> I["展示语聊房最小化样式，而不是 Games 首页样式"]

    I --> J{"后续操作"}
    J -->|"点击语聊房最小化浮窗"| K["恢复该语聊房"]
    K --> L{"游戏是否仍在进行？"}
    L -->|"是"| M["房内继续展示游戏最小化入口，可恢复 H5"]
    L -->|"否"| N["回到普通语聊房状态，不展示游戏入口"]

    J -->|"点击其他 Game Room / 推荐游戏"| O["弹窗：确认离开当前房间么？"]
    J -->|"点击 Private room"| P["弹窗：确认离开当前房间么？"]
    O --> Q{"确认离开当前房间么？"}
    P --> Q
    Q -->|"确认"| R["退出当前房间；若本人为游戏玩家，同步执行普通用户退出游戏逻辑"]
    Q -->|"取消"| S["留在当前语聊房最小化状态"]
```

**规则说明：**
- Discover 只是入口来源；进入别人游戏房后，用户已经处在某个房主的语聊房内。
- 游戏最小化只回到该房间语聊界面；房间再次最小化时，复用平台现有语聊房最小化样式。
- 后续恢复、退出、切房等操作都按“当前所在语聊房 + 普通用户游戏身份”处理，不回到 Games 首页承接。

**规则说明：**
- 平台现有逻辑为：房主离开房间后，房间不会立刻关闭；其他用户仍可继续在语聊房内聊天、送礼。
- 房主在 playing 中离房时，不强制关闭整局；房主本人的游戏角色由系统接管，其他已加入游戏/观战用户可完成本局。
- 本局结束后，所有用户退回语聊房状态，并且不再展示该游戏入口，避免误以为房主仍在维护游戏房。
- 未加入游戏且未观战的用户，不需要看到游戏最小化入口；他们只保留普通语聊房体验。

### 13.9 WeChill 优化后完整路径总结

**路径 1：Discover → 快速匹配（最快路径，参考Hawa）**

**路径 2：Discover → 选择Game Room（稳定路径，原有方案优化）**
```
Discover → 点击游戏卡片 → 游戏详情聚合页 → Active Rooms列表 → 选择房间 → 进入(全屏H5)
```

**路径 3：Private room（房主自开，原有方案+配置优化）**
```
Private room → 进入自己房间 → Games面板 → 选游戏 → 极简配置(Mode+Bet+Match+AutoNext) → Start → Loading → 全屏H5
```

**路径 4：房内Games面板（房主已在房间，原有方案）**
```
已在自己房间 → Games面板 → 选游戏 → 极简配置 → Start → 全屏H5
```

**路径 5：Home页发现游戏房（新增，参考Hawa）**
```
Home → 看到带游戏Badge的房间卡片 → 点击 → 进入游戏房(全屏H5)
```

**路径 6：游戏内循环（新增，参考Hawa Auto-next）**
```
游戏结束 → 结果卡片(7s倒计时) → 自动下一局 → Loading → 继续游戏 → ...循环
```
