## 1. 背景与目标

### 1.1 背景

WeChill 语聊房当前以语音社交为主，用户停留和互动方式相对单一。引入三方休闲游戏可以：

- 增加房间停留时长和互动深度。
- 提升付费转化和金币消耗场景。
- 降低用户社交冷启动门槛。

本期主路径仍是"语聊房内直接调起游戏"。同时保留 Discover `Private room` 快捷入口：用户可在 Discover 当前页先选择休闲游戏，系统随后进入用户自己的房间并打开该游戏的设置页，由房主确认人数 / 模式 / 门票后创建游戏房并进入等待态；首次进入时不先展示语聊房主界面。

**核心业务逻辑（用户补充）：**

- 在 WeChill 平台里，每个用户都可以是房主（在自己房间内就是房主）。
- 游戏房 = 游戏所在的某人的房间。
- 其他所有人都是普通用户。
- Discover 页面选择时，第一层判断：用户是否在房间（一定是最小化状态）。
- 在自己房间 → 先退出房间 → 以普通用户身份匹配游戏房。
- 在别人房间 → 退出别人房间 → 再执行目标操作。
- 点击 Private room → 进入自己房间 → 开启游戏房模式。

### 1.2 目标

| 目标 | 说明 |
|---|---|
| 快速验证 | 不改造房间模型，在现有语聊房内直接调起三方 H5 游戏 |
| 降低上线风险 | 不改房间类型，控制改造范围，支持灰度和下架 |
| 适配 MENA | 优先接入 Ludo、UNO、Domino、8 Ball、Carrom、Snake & Ladder |
| 快速体验 | Discover `Private room` 支持当前页选游戏、自动进入自己的房间并打开设置页；推荐游戏支持 15s 匹配等待，降低找局成本 |
| 资金安全 | 门票采用预扣（冻结）+确认方案，H5 加载失败或超时自动退还 |

## 3. 用户角色与权限

### 3.1 房间归属口径

WeChill 平台内每个用户都有自己的语聊房。本文中的"房主"不是平台级固定身份，而是**当前房间的归属用户**：

- 用户进入自己的房间时，该用户就是当前房间房主，可以开启游戏、关闭游戏、调整局参数。
- 用户进入其他人的房间时，无论其是否拥有自己的房间，在该房间内都按普通用户处理。
- Discover 页面进入 Game Rooms 或推荐游戏时，目标是进入"某个用户的房间里的游戏"，当前用户进入后默认是普通用户。
- Discover `Private room` 是特殊快捷入口：用户进入自己的房间，并直接开启游戏房模式。

### 3.2 角色定义

| 角色 | 判定方式 | 说明 |
|---|---|---|
| 当前房间房主 | `room.owner_user_id == current_user_id` | 在自己房间的用户就是房主；可开启游戏、关闭游戏、调整局参数；waiting/ready 可管理座位，playing 中不允许踢出游戏中玩家 |
| 普通用户 | `room.owner_user_id != current_user_id` | 进入别人房间的所有人；可查看、加入、观战、送礼、聊天 |
| 观战状态 | RoomGamePlayer.status=watching | 普通用户的一种状态，不是独立角色；可观看游戏、公屏聊天、送礼，不可操作游戏 |
| 审核账号 | 后台策略 | 可隐藏入口、隐藏付费局和游戏币兑换入口，只展示低风险内容 |
| 被封禁用户 | 风控态 | 不展示入口，不允许进入游戏 |

说明：

- V1 游戏控制权只归当前房间房主。若平台存在房管 / 管理员能力，V1 在游戏内仍按普通用户处理，不开放 Start / Close / Kick Seat。
- Discover 匹配到游戏房后，用户一定是进入他人的房间，按普通用户状态 Join / Watch / Open。
- Private room 进入的是用户自己的房间，用户是当前房间房主；选游戏后进入设置页，确认配置后创建 RoomGameSession。
- 观战是普通用户的一种状态（RoomGamePlayer.status=watching），不需要独立角色。
- 不存在管理员接管房间或转移房主的概念。playing 中房主退出 / 关闭游戏时，仅由系统接管房主在本局里的游戏角色，其他玩家继续完成本局；本局结束后不再展示游戏入口。

### 3.3 权限矩阵

| 功能 | 当前房间房主 | 普通用户 | 观战中 | 审核账号 |
|---|---|---|---|---|
| 查看游戏入口 | 是 | 是 | 是 | 按策略 |
| 打开游戏面板 | 是 | 是 | 是 | 按策略 |
| 开启游戏 | 是 | 否 | 否 | 否 |
| 关闭游戏 | 是 | 否 | 否 | 否 |
| 加入游戏座位 | 是（开局时自动加入） | 是（Join） | 仅在 waiting 状态可 Join | 否 |
| 观战 | 是 | 是 | — | 按策略 |
| 踢出游戏座位 | 仅 waiting/ready 可管理；playing 不允许踢出游戏中玩家 | 否 | 否 | 否 |
| 配置游戏（人数/模式/门票） | 是 | 否 | 否 | 否 |
| 邀请他人 | 是 | 否（V1） | 否 | 否 |
| 最小化游戏 | 是 | 是 | 是 | 按策略 |
| 再来一局 | 是 | 可 Join | 可 Join | 按策略 |

补充说明：
- 房主 = 游戏的 game_controller，拥有完整的游戏管理权限。
- 普通用户只能 Join / Watch / 退出，不能管理游戏。
- V1 中途补位默认不支持：观战用户只能在 waiting 状态转为 joined。
- 房主主动退出房间 = 游戏房关闭；房主异常断线按 14.2 分层处理。

- 门票在 Start 时预扣（frozen），gameLoaded 成功 + type=13 收到后确认扣费（success），任何环节失败自动退还（refunded）。

### 6.6 游戏最小化、入口保留与恢复

> 本节整合自 v1.5 全角色最小化交互场景，并统一 Private room 与房内入口的最小化路径。

游戏最小化的核心原则：**最小化后只保留游戏入口，不展示麦位信息；再次打开游戏时，必须重新拉取并同步房间最新麦位、语音、座位和游戏状态。**

Private room 只是 Discover 侧的快捷入口，不是独立的游戏返回路径。用户从 Private room 选择游戏后，系统已经进入用户自己的房间并创建 RoomGameSession；因此游戏最小化、恢复、关闭和结算都以自己的房间为返回上下文，不回到 Discover Games 首页。用户从 Discover 推荐游戏或 Game Rooms 进入时，返回上下文是匹配到的目标游戏房，用户在该房间内仍是普通用户。

若用户从 Discover 进入别人的游戏房，游戏中点击最小化只收起 H5 并回到该房间语聊界面；如果用户随后再次最小化房间，走平台原有语聊房最小化逻辑，展示语聊房最小化样式，而不是回到 Games 首页或展示游戏首页样式。此后用户点击其他 Game Room / 推荐游戏 / Private room 时，统一弹窗 `确认离开当前房间么？`，确认后再退出当前房间并处理个人游戏态。

```mermaid
flowchart TD
  A["用户点击游戏最小化"] --> B["App 调用 gameActionUpdate type=2016 status=minimize"]
  B --> C["App 收起 H5 游戏容器"]
  C --> D["回到 RoomGameSession 所属语聊房主界面"]
  D --> E{"入口来源"}
  E -->|"房内入口"| F["回到原语聊房"]
  E -->|"Private room"| G["回到自己的房间"]
  E -->|"Discover 匹配游戏房"| R["回到匹配到的目标游戏房"]
  F --> H["房间展示完整原生麦位"]
  G --> H
  R --> H
  H --> I["展示游戏最小化入口"]
  I --> J["入口只展示游戏名/状态/玩家数/Open，不展示麦位"]
  J --> K["用户点击 Open"]
  K --> L["先发送 gameActionUpdate type=2016 status=expand"]
  L --> M["等待 H5 恢复渲染（建议 500ms）"]
  M --> N["查询 RoomGameSession 最新状态"]
  N --> O["查询 RoomMicState 最新麦位"]
  O --> P["查询 RoomGamePlayer 最新游戏座位"]
  P --> Q["查询 RTC 静音/说话状态和钱包余额"]
```

| 信息 | 是否展示 | 说明 |
|---|---|---|
| 游戏名称 | 展示 | 例如 Ludo / UNO |
| 游戏状态 | 展示 | waiting / playing / settling |
| 游戏人数 | 展示 | 例如 2/4 players |
| 恢复按钮 | 展示 | Open / Continue |
| 麦位头像 | 不展示 | 避免和房间原生麦位重复 |
| 麦克风状态 | 不展示 | 最小化入口不承载语音状态 |
| 说话动效 | 不展示 | 统一由房间主界面展示 |
| 游戏座位详情 | 不展示 | 点击进入游戏后再展示 |


恢复游戏时必须重新同步的数据：

| 数据 | 数据源 | 用途 |
|---|---|---|
| RoomGameSession | 我方游戏会话服务 | 判断游戏是否仍在进行、是否已结算、是否异常 |
| RoomMicState | 我方语聊房服务 | 渲染游戏内麦位缩略条 |
| RoomGamePlayer | 我方游戏座位服务 + Bytesun 回调 | 恢复玩家座位 / 观战状态 |
| RTC 状态 | 我方语音服务 | 同步静音、禁麦、说话状态 |
| wallet balance | 我方钱包服务 | 刷新游戏币余额 |
| room permission | 我方房间权限服务 | 判断用户是否仍有进入房间和游戏的权限 |

恢复时的异常处理：

| 场景 | 处理 |
|---|---|
| 游戏仍在 playing | 直接恢复游戏容器，并按最新 RoomMicState 渲染麦位缩略条 |
| 游戏已 ended | 不恢复 H5，提示"当前游戏已结束"，可展示结算结果 |
| 游戏 settling 中 | 恢复结算中页面，不允许重复操作 |
| 用户已被踢出房间 | 未参游/观战用户不允许恢复游戏；playing 玩家默认不允许被踢出游戏，按 B-08 待确认 |
| 用户游戏座位已被释放 | 恢复为 Watch / Join 状态 |
| 麦位状态已变化 | 以最新 RoomMicState 为准刷新，不使用最小化前缓存 |
| H5 已被系统回收 | 重新加载 H5，并通过 getConfig 返回最新状态 |

客户端实现要求：

- 游戏最小化入口只作为"恢复游戏的入口"，不承担麦位展示职责。
- 房间主界面已经有完整麦位时，游戏入口不得重复展示麦位头像或说话状态。
- 用户点击 Open 恢复游戏时，客户端不得直接使用最小化前的麦位缓存。
- 恢复游戏前必须先拉取 RoomMicState，再渲染游戏内麦位缩略条。
- 如果 RoomMicState 拉取失败，游戏可先恢复，但麦位缩略条展示 loading / retry，不展示旧数据。
- 用户最小化期间发生上麦、下麦、锁麦、禁麦、踢出房间等变化，恢复游戏后必须同步为最新状态。
- 最小化时发送 `gameActionUpdate type=2016 status=minimize` 通知 H5，恢复时发送 `type=2016 status=expand`。

### 6.7 全角色游戏最小化交互规则

> 本节沿用 v1.5 的全角色最小化交互矩阵，与 14.3 结构化规则互补。

所有已进入游戏容器的用户，包括当前房间房主、普通玩家和观战用户，均可最小化游戏。

**最小化仅代表收起游戏容器，不代表退出游戏、不代表退出房间、不释放游戏座位，也不改变房间麦位状态。**

| 角色 / 身份 | 是否可最小化 | 最小化后状态 | 再次打开后的处理 |
|---|---:|---|---|
| 当前房间房主 | 是 | 游戏继续，房主身份和游戏控制权不变 | 重新同步 session、座位、麦位、RTC、钱包和权限 |
| 普通玩家 | 是 | 游戏座位保留，当前局继续 | 恢复到玩家态；若已被移出座位，则展示 Watch / Join / Removed |
| 观战用户 | 是 | 观战关系保留，不影响玩家 | 游戏仍在进行则恢复观战，已结束则展示结束态或隐藏入口 |
| 未进入游戏的房间用户 | 不属于"最小化" | 可看到当前游戏快捷入口 | waiting 展示 Join，playing 展示 Watch / Open |
| 被踢出游戏的用户 | 否 | 不展示最小化入口 | 可按状态展示 Watch / Join 或移除入口 |
| 被踢出房间的用户 | 否 | 未参游/观战用户清理游戏入口；playing 玩家默认不允许被踢出游戏 | 未参游/观战不允许恢复游戏；playing 按 B-08 待确认 |

说明：

- "进入过游戏后再收起"叫游戏最小化入口。
- "未进入游戏但房间有游戏"叫当前游戏快捷入口。
- 两者可以复用 UI 样式，但埋点、恢复逻辑和按钮文案需要区分。

### 6.8 不同角色的最小化交互场景

#### 6.8.1 房主最小化

| 游戏状态 | 房主最小化后 | 房主再次打开后 |
|---|---|---|
| waiting | 游戏等待态保留，其他用户仍可 Join | 可 Start / Invite / Kick Seat / Close |
| ready | 准备态保留 | 可 Start / Kick Seat / Close |
| playing | 本局继续 | 恢复游戏画面，继续管理异常 |
| settling | 结算继续 | 展示结算中，不允许重复操作 |
| ended | 浮窗可展示 Finished 或隐藏 | 可看结果 / 再来一局，按后台配置 |

房主最小化不等于关闭游戏。关闭当前游戏必须通过游戏展开态内的关闭入口，且需按权限和游戏状态判断。

#### 6.8.2 普通用户最小化（含原房管/管理员）

> V1 中房管/管理员在游戏内按普通用户处理，不展示管理权限。

| 场景 | 处理 |
|---|---|
| 只是观战 | 按普通观战用户处理，最小化后回房间，点击 Open 恢复观战 |
| 是玩家 | 按普通玩家处理，保留游戏座位，恢复后回到玩家态 |
| 房主在其最小化期间离开 | 按 14.2 主动退出 / 异常断线分层规则处理 |

#### 6.8.3 普通玩家最小化

普通玩家可以最小化游戏，但需要兼容回合制游戏的实时操作风险。

| 状态 | 最小化入口展示 | 点击后 |
|---|---|---|
| waiting | Waiting | 打开等待页 |
| playing | Playing | 恢复游戏 |
| 轮到自己操作 | Your turn / Playing | 恢复游戏并定位到当前操作态 |
| 即将超时 | Hurry up / Playing | 恢复游戏并展示倒计时，若三方不支持则不展示倒计时 |
| settling | Settling | 打开结算中页 |
| ended | Finished | 展示结果或隐藏入口 |

V1 如果 Bytesun 暂不支持"轮到你 / 倒计时"状态回调，最小化入口只展示 Waiting / Playing / Settling / Finished / Error。

#### 6.8.4 观战用户最小化

| 游戏状态 | 观战用户最小化后 | 再次打开后 |
|---|---|---|
| waiting | 入口展示 Waiting / Join | 有座位可 Join，无座位可 Watch |
| playing | 入口展示 Watching / Playing | 恢复观战 |
| settling | 入口展示 Settling | 展示结算中或观战结束态 |
| ended | 入口展示 Finished 或隐藏 | 可展示结果摘要，或提示游戏已结束 |

V1 默认不支持中途补位。观战用户在 playing 中恢复后默认仍为观战态，除非三方明确支持中途加入。

### 6.9 最小化入口状态与点击行为

| 状态 | 浮窗文案 | 入口按钮 | 点击行为 |
|---|---|---|---|
| waiting | Waiting | Open / Join | 打开游戏等待页 |
| ready | Ready | Open | 打开准备页 |
| playing | Playing | Open | 恢复游戏容器 |
| watching | Watching | Open | 恢复观战页 |
| your_turn | Your turn | Play | 恢复游戏并高亮当前操作 |
| timeout_warning | Hurry up | Play | 恢复游戏并展示倒计时 |
| settling | Settling | Open | 打开结算中页 |

```text

```

不展示麦位头像、游戏座位详情、麦克风状态和说话动效。

### 6.10 最小化恢复顺序

用户点击 Open 后，客户端不得直接恢复旧 H5 画面并使用缓存状态，必须按以下顺序恢复：

```mermaid
flowchart TD
  A["点击最小化入口 Open"] --> B["校验用户是否仍在房间"]
  B --> C{"房间是否有效"}
  C -->|"无效"| D["清理入口，提示房间已结束或已被移出"]
  C -->|"有效"| E["查询 RoomGameSession"]
  E --> F{"游戏 session 是否有效"}
  F -->|"无效 / closed"| G["隐藏入口或展示 Finished"]
  F -->|"有效"| H["查询 RoomGamePlayer 用户游戏身份"]
  H --> I["查询 RoomRoleState，判断是否为当前房间房主"]
  I --> J["查询 RoomMicState 最新麦位"]
  J --> K["查询 RTCState 静音/禁麦/说话状态"]
  K --> L["查询 WalletBalance 游戏币余额"]
  L --> M["发送 gameActionUpdate type=2016 status=expand"]
  M --> N["等待 500ms H5 恢复渲染"]
  N --> O["恢复或重建 H5 游戏容器"]
  O --> P["按最新状态渲染游戏页和麦位缩略条"]
```

恢复优先级：

1. 先判断用户是否仍在房间。
2. 再判断游戏 session 是否有效。
3. 再判断用户是玩家、观战、被移出还是未加入。
4. 再同步房间权限、麦位、RTC、钱包。
5. 发送 type=2016 expand 通知 H5 恢复渲染。
6. 最后恢复 H5 容器。

### 6.11 最小化期间状态变化兼容

| 最小化期间发生的变化 | 浮窗表现 | 用户点击 Open 后 |
|---|---|---|
| 用户被抱上麦 | 浮窗不展示麦位变化 | 游戏内麦位缩略条展示最新麦位 |
| 用户被移下麦 | 浮窗不展示麦位变化 | 游戏内不再展示该用户在麦上 |
| 用户被禁麦 | 浮窗不展示禁麦 | 游戏内麦位缩略条展示禁麦状态 |

| 用户被移出游戏座位 | 浮窗可变为 Removed / Watch | 展示已移出座位，按状态允许 Watch / Join |
| 用户被踢出房间 | 未参游/观战浮窗消失；playing 玩家默认不允许被踢出游戏 | 未参游/观战不允许恢复游戏；playing 按 B-08 待确认 |
| 房主离开 | 普通用户浮窗不变 | 按 14.2 主动退出 / 异常断线分层规则处理 |

离开房间确认文案：

```text
你正在游戏中，进入其他房间将退出当前游戏，是否继续？
```

### 6.12 入口进入游戏的完整路径

> 本章节基于正确的角色模型完整定义入口逻辑。核心原则：
> - **房主** = 用户在自己的房间里，就是房主
> - **普通用户** = 进入别人房间的所有人
> - **Private room** = 进入自己的房间 + 开启游戏房模式

#### 6.12.0 角色模型与核心逻辑

**角色定义：**

| 角色 | 定义 |
|---|---|
| 房主 | 用户在自己的房间 = 房主。每个用户都有自己的房间 |
| 普通用户 | 进入别人的游戏房的人，无论什么身份 |

**Discover 入口第一层判断逻辑：**

用户在 Discover 页面操作时，必须先判断当前是否在某个房间（房间一定是最小化状态的）：
1. 不在任何房间 → 直接执行目标操作
2. 在自己的房间 → 提示退出自己房间 → 以普通用户身份去匹配游戏房
3. 在别人的房间 → 提示退出别人房间 → 再执行目标操作

| Discover 推荐游戏 | 普通用户 | 不在任何房间（否则先退出） | 选 Game Room → 进入房间 → Join/Watch |
| Discover Game Rooms | 普通用户 | 同上 | 进入房间 → Join/Watch |
| 别人房间内浮窗 | 普通用户 | 已在别人房间 | Join/Watch |
| 邀请链接 | 普通用户 | 不在其他房间（否则先退出） | 进入房间 → Join/Watch |
| 审核账号 | — | 按后台策略 | 隐藏入口 或 只展示免费游戏 |

### 6.13 门票预扣与确认扣费时序

> 本章节统一门票扣费时机，解决 v1.6 中"type=13 时扣费"的资金风险。

#### 6.13.1 统一方案：预扣（冻结）+ 确认

```mermaid
flowchart TD
    A["用户点击 Start / Join"] --> B["服务端校验权限和座位"]
    B --> C{"校验通过"}
    C -->|"否"| D["返回错误码"]
    C -->|"是"| E["预扣门票（冻结游戏币）"]
    E --> F{"预扣成功"}
    F -->|"游戏币不足 (1008)"| G["弹窗 → 跳转 Game coins 兑换页"]
    F -->|"成功"| H["生成 ticket_order (status=frozen)"]
    H --> I["加入/创建 RoomGameSession"]
    I --> J["加载 H5 → 等待 gameLoaded"]
    J --> K{"gameLoaded 是否成功（30s 超时）"}
    K -->|"成功"| L["等待 type=13（游戏开始）"]
    K -->|"超时/失败"| M["ticket_order.status = refunded"]
    M --> N["通知用户：游戏启动失败，已退回门票"]
    L --> O{"收到 type=13"}
    O -->|"收到"| P["ticket_order.status = success（确认扣费）"]
    O -->|"60s 超时未收到"| Q["ticket_order.status = refunded"]
    Q --> N
    P --> R["RoomGameSession = playing"]
```

#### 6.13.2 各阶段门票状态

| 阶段 | ticket_order.status | 用户余额 | 退款条件 |
|---|---|---|---|
| 点击 Start/Join 预扣 | frozen | 已冻结（余额减少） | — |
| gameLoaded 成功 | frozen | 继续冻结 | 加载失败自动退 |
| type=13 收到（游戏开始） | success | 正式扣除 | 不可退 |
| 加载失败 / 超时 | refunded | 退还冻结金额 | 自动退 |
| type=13 超时（60s） | refunded | 退还 | 自动退 |
| playing 中异常结束 | 视结算结果 | 按结算处理 | 按三方规则 |

#### 6.13.3 预扣超时与余额展示

- 预扣总超时：gameLoaded 30s + type=13 60s = 总计 90s
- 冻结期间的余额展示：建议在游戏中展示"可用余额 = 总余额 - 冻结金额"

### 6.14 房间系统消息与游戏召回

参考 Hawa 的“房主创建游戏后自动发送房间消息”能力，但 WeChill V1 控制频次，避免刷屏。

| 场景 | 触发 | 展示 | 点击行为 |
|---|---|---|---|
| 房主创建游戏成功 | RoomGameSession 从 loading 进入 waiting | 房间系统消息：“房主开启了 Ludo，快来加入” | 打开当前游戏入口 |
| 普通用户加入游戏 | RoomGamePlayer.status=joined | 可选轻提示，不刷公屏 | 无 |
| 游戏开始 | 收到 type=13 | 房间系统消息：“Ludo 已开始，可观战” | Watch / Open |
| 游戏结束 | 收到 type=14 且结算完成 | 房间系统消息展示结果摘要 | 查看结果 |
| 最小化召回 | 用户离开游戏容器但 session 有效 | 浮窗文案展示游戏名 + 状态 + 人数 | 恢复 H5 |

系统消息频控：

- 同一 RoomGameSession 的“创建游戏成功”消息只发送一次。
- 房主短时间重复失败重试不重复刷房间消息。
- 审核账号、封禁账号、地区不可见游戏不展示召回消息。
- 系统消息不改变游戏状态，点击后仍按 RoomGameSession 最新状态判断 Join / Watch / Open。

## 7. 游戏状态设计

### 7.1 RoomGameSession 状态机

```mermaid
stateDiagram-v2
  [*] --> none
  none --> selecting: 打开游戏面板
  selecting --> loading: 选择游戏并启动
  loading --> waiting: gameLoaded
  waiting --> ready: 玩家准备完成
  ready --> playing: type=13
  playing --> settling: type=14
  playing --> reconnecting: 网络异常
  reconnecting --> playing: 恢复成功
  reconnecting --> settling: 超时或已结束
  settling --> ended: 结算完成
  ended --> waiting: 再来一局
  ended --> none: 关闭游戏
```

### 7.2 状态定义

| 状态 | 含义 | 用户可见动作 |
|---|---|---|
| none | 当前房间无游戏 | 打开游戏面板 |
| selecting | 浏览游戏列表 | 选择游戏 |
| loading | H5 加载中 | 等待 / 取消 |
| waiting | 游戏已创建，等待用户加入 | Join / Invite |
| ready | 达到开局条件 | Start |
| playing | 游戏进行中 | Open / Watch / Gift / Chat |
| reconnecting | 重连中 | 等待 / 退出 |
| settling | 结算中 | 等待 |
| ended | 已结束 | 再来一局 / 返回房间 |
| error | 异常 | Retry / Close |

## 8. 座位、麦位与声音规则

| 规则 | 说明 |
|---|---|
| 游戏座位独立于语聊麦位 | 用户不上麦也可以玩游戏 |
| 语聊麦位不因游戏改变 | 原有房间归属、主持、房管等语聊房身份保持；V1 游戏控制权只看当前房间归属用户 |
| 座位以 App 最终状态为准 | 与 H5 冲突时，通过 gameActionUpdate type=4 反向同步 |
| 一个用户只能占一个游戏座位 | 防重复占位和多端异常 |
| 游戏中不可随意换座 | 除非三方游戏明确支持 |
| 游戏结束释放座位 | 再来一局重新进入 waiting / preparing |

声音策略：

| 场景 | 策略 |
|---|---|
| 普通休闲游戏 | 保留我方语聊房 RTC，默认降低或关闭游戏 BGM |
| 用户静音麦克风 | 同步我方 RTC 状态，必要时通知游戏 |
| 游戏查询音效状态 | H5 发查询后，App 返回当前 sound 状态 |
| 游戏内音频播放（type=3003） | 按背景音乐/音效类型分别处理，与语聊房 RTC 不冲突 |
| 麦克风/扬声器状态同步（type=3001） | H5→App 和 App→H5 双向同步 |
| RTC 推理游戏 | V3 才接入，使用 isGameRTC=true 和 type=3001 |

## 9. 货币与结算

### 9.1 货币口径

V1 游戏内统一使用**游戏币（Game coins）**，金币只作为兑换来源，不直接参与游戏下注、门票和结算。

```text
金币 Gold coins -> 兑换 -> 游戏币 Game coins -> 游戏内门票/下注/结算
```

默认兑换比例按页面展示：

```text
100 金币 = 1000 游戏币
```

规则：

| 规则 | 说明 |
|---|---|
| 游戏内展示币种 | 只展示游戏币，不展示金币 |
| 游戏币来源 | 用户通过金币兑换获得 |
| 反向兑换 | 不支持游戏币兑换回金币 |
| 提现 | 游戏币仅供游戏内使用，不支持提现 |
| 比例配置 | 默认 100 金币 = 1000 游戏币，后台可配置并记录历史版本 |
| 审核策略 | 审核账号可隐藏付费局和游戏币兑换入口 |

### 9.2 消耗与奖励场景

| 场景 | 类型 | 币种 | 变动 |
|---|---|---|---:|
| 金币兑换游戏币 | exchange | 金币 / 游戏币 | 金币负值，游戏币正值 |
| 门票预扣 | ticket_freeze | 游戏币 | 负值（冻结） |
| 门票确认扣费 | ticket | 游戏币 | 冻结转正式扣除 |
| 游戏下注 | bet | 游戏币 | 负值 |
| 游戏结算奖励 | result | 游戏币 | 正值 |
| 异常退款 | refund | 游戏币 | 正值 |
| 运营补偿 | compensate | 游戏币 | 正值 |

### 9.3 游戏币不足处理

| 触发点 | 判断 | 处理 |
|---|---|---|
| 设置页点击 Start / Join | 游戏币 < 门票 | 弹窗提示"游戏币不足"，跳转 Game coins 兑换页 |
| 门票预扣失败 | 服务端返回余额不足 | 返回 1008，弹窗引导兑换 |
| H5 内下注 | Bytesun 调 change_balance 返回 1008 | H5 调 `gameRecharge`，App 打开 Game coins 兑换页 |
| 上座校验 | 游戏币低于当前局最低要求 | 不允许上座，提示兑换后重试 |
| 兑换页金币不足 | 金币余额 < 需扣金币 | 兑换按钮置灰，引导充值或获取金币 |
| 兑换成功 | 金币扣减成功，游戏币增加成功 | 调 `walletUpdate` 通知 H5 刷新余额，并允许回到原游戏 |

### 9.4 结算安全

| 规则 | 说明 |
|---|---|
| order_id 幂等 | 同一订单重复回调只处理一次 |
| 用户级锁 | change_balance 必须做单用户并发保护 |
| 成功 code | 只有成功才能返回 code=0 |
| 游戏币不足 | 返回 1008，H5 可调 gameRecharge |
| 兑换幂等 | 金币扣减和游戏币增加必须在同一事务内完成 |
| 门票预扣幂等 | ticket_order 按 order_id 幂等，不重复冻结 |
| 预扣超时自动退还 | gameLoaded 30s + type=13 60s 超时后自动 refunded |
| 补偿队列 | 结算失败进入补偿和对账 |
| 对账 | 游戏订单、兑换订单、钱包流水、Bytesun 回调四方核对 |

## 10. Bytesun 技术接入要求

### 10.1 接入方式

| 方式 | 说明 | V1 建议 |
|---|---|---|
| URL 直连 | 直接加载游戏 H5 URL | 联调和灰度优先 |
| Zip 本地包 | 下载游戏 Zip 并解压本地加载 | 正式体验优化 |
| CDN 加速 | OSS 源站回源，配置 CDN | 必做 |

标准流程：

1. 我方后台同步 Bytesun 游戏列表。
2. 客户端获取游戏信息和版本。
3. 客户端按版本决定 URL 直连或 Zip 更新。
4. 打开 H5 / WebView。
5. H5 调 `getConfig`。
6. App 返回商户、用户、房间、语言、游戏币余额、金币余额、角色等配置。
7. H5 通过 JSBridge 上报游戏状态。
8. Bytesun 服务端通过我方 API 查询用户和修改余额。

### 10.2 getConfig 关键字段

| 字段 | 说明 |
|---|---|
| merchantId | Bytesun 分配 |
| appChannel | Bytesun 分配，后台配置 |
| userId | 我方用户 ID |
| roomId | 当前语聊房 ID |
| role | 用户角色（0=普通用户，1=观战，2=房主） |
| language | 客户端语言（映射为 Bytesun language 值） |
| gameMode | 语聊房场景固定传 `"3"` |
| balance | 当前游戏币余额，供 Bytesun 游戏内展示和扣费 |
| goldBalance | 当前金币余额，仅用于我方兑换页展示，不直接参与游戏扣费 |
| exchangeRate | 金币兑换游戏币比例，例如 `100金币=1000游戏币` |
| userType | 用户风控 / 审核类型 |
| safeArea | 顶部 / 底部安全区 |

## 14. 异常与边界场景

本章节补充房间生命周期、房主退出、最小化、座位、麦位、H5 状态、钱包结算、后台干预、审核风控等异常场景。整体原则：

| 原则 | 说明 |
|---|---|
| 房间是主容器 | 游戏依附于语聊房，不脱离房间单独运行 |
| 游戏是插件态 | 游戏不反向改变语聊房归属、麦位和房管体系；V1 游戏控制权只看当前房间归属用户 |
| 服务端状态为准 | RoomGameSession、RoomGamePlayer、钱包订单为最终依据 |
| 已开局优先不中断 | playing / settling 状态优先保障本局完成和结算稳定；房主主动退出或关闭游戏时，不强制中断其他玩家本局 |
| 钱包订单必须幂等 | 扣费、退款、结算、补偿均需订单关联和幂等处理 |
| 不做房间接管 | 不存在管理员接管房间、转移房主或无主游戏房；playing 中仅允许系统接管房主本局游戏角色 |

### 14.1 房间相关

| 场景 | 处理 |
|---|---|
| 当前房间房主主动退出房间 | waiting / ready 关闭 session 并退款；playing 中房主本人退出游戏，房主游戏角色由系统接管，其他玩家 / 观战者完成本局；本局结束后不展示入口 |
| 当前房间房主异常断线 | 0-120s 保留重连；超时后 waiting / ready 关闭并退款；playing / settling 可由 Bytesun 侧完成本局，结束后关闭 session |
| 房间关闭 | 强制结束游戏，按状态结算或退款 |
| 房间被封禁 | 立即关闭游戏入口和当前游戏层，当前局按风控策略结算、挂起或退款 |
| 用户被踢出房间 | 未加入/未观战用户按语聊房逻辑离房；观战用户退出观战；游戏中玩家默认不允许被房主踢出游戏，B-08 待业务确认 |
| 房间切后台 | 游戏继续，前台回来恢复状态 |
| 多人同时开游戏 | 服务端只允许一个 session，后发请求失败 |
| 用户游戏中进入其他房间 | 二次确认；确认后退出当前游戏并释放座位 |
| 房间改为私密 / 加锁 | 已在房间用户不受影响，新用户按新权限校验 |
| 游戏中切换房间模式 | playing 中不允许普通切换，后台强制切换时游戏继续但权限按新模式生效 |
| 房间无人 | waiting 关闭 session；playing 按三方规则完成、托管、结束或退款 |

### 14.2 当前房间房主退出与断线处理

> **v2.0 关键裁决：** 不做房间管理员接管，也不转移房主身份。若游戏已进入 playing，房主主动退出房间或主动关闭游戏时，房主本人按普通用户退出游戏逻辑处理，房主的本局游戏角色由系统接管，其他玩家 / 观战者继续完成本局；本局结束后退回语聊房状态，不展示游戏入口。waiting / ready 未开局状态仍可关闭 session 并退款。

#### 14.2.1 核心原则

**房主退出不等于关闭语聊房。** 平台现有逻辑为：房主离开房间后，房间不会立刻关闭，其他用户仍可继续在语聊房内聊天、送礼。因此游戏处理需要拆分：

- waiting / ready：未开局，可关闭当前 RoomGameSession，退还已预扣门票。
- playing：不强制中断其他已付费玩家本局；房主本人退出游戏，房主本局游戏角色由系统接管。
- settling：不打断结算；结算完成后关闭游戏入口。

不存在“房间接管”概念：
- 不存在管理员接管房间。
- 房主身份不转移。
- 不存在无主游戏房继续开放新局。
- “系统接管”仅指系统接管房主在当前本局里的游戏角色，用于保障本局完成。

异常断线不是主动退出。为避免误杀本局和结算，断线仍按 14.2.2 分层处理。

#### 14.2.2 房主断线分层处理

```mermaid
flowchart TD
    A["房主网络异常/断线"] --> B{"断线时长"}
    B -->|"0-30s"| C["标记 reconnecting，游戏继续"]
    B -->|"30-120s"| D["标记离线中，游戏暂停/继续（取决于三方规则）"]
    D --> E["普通用户可见房主断线提示"]
    B -->|">120s"| F["判定房主离开"]
    
    F --> G{"当前游戏状态"}
    G -->|"waiting/ready"| H["关闭 session，所有门票退款"]
    G -->|"playing"| I["允许本局继续（由三方游戏托管），本局结束后自动关闭"]
    G -->|"settling"| J["结算继续完成，完成后关闭"]
    
    H --> K["通知所有普通用户：游戏已关闭"]
    I --> L["本局结束后 → 关闭游戏入口"]
    J --> L
    K --> M["房间回到普通语聊房状态"]
    L --> M
```

**分层处理明细：**

| 断线时长 | 处理 |
|---|---|
| 0-30s | 保留游戏座位，标记 reconnecting，游戏继续，用户无感恢复 |
| 30-120s | 继续保留座位，房间内标记离线中，普通用户可见断线提示 |
| >120s | 判定房主离开，按游戏状态关闭或允许本局继续后关闭 |

**playing 中房主断线的注意事项：**
- WeChill 侧不维护房间级 system 托管状态。
- playing 中房主断线后，WeChill 只做：1）通知 Bytesun 该玩家断线；2）本局结束后关闭 session；3）不允许再开新局。
- "三方游戏托管"是指 Bytesun 游戏侧的自动运行（如 Ludo 断线后 AI 代打），不是 WeChill 侧的 system 托管。
- 普通用户在本局期间仍可继续游戏。

```mermaid
    O --> P["游戏玩家 / 观战用户退出当前游戏，退回语聊房"]
    P --> Q["不展示游戏入口 / 游戏最小化入口"]
    K --> R["房间恢复普通语聊房状态"]
    Q --> R
    R --> S["房主退出房间；其他用户仍可继续聊天 / 送礼"]
```

#### 14.2.4 与旧版"接管"逻辑的对比

| 旧版设计 | 新版设计 | 原因 |
|---|---|---|
| 房主离开 → 管理员接管 | 不做房间接管；playing 中仅系统接管房主本局游戏角色 | 保留房间二角色体系，不引入管理员游戏控制权 |
| 房主离开 → 立即强制关闭 playing 游戏 | 不强制中断本局，其他玩家继续完成 | 保护已付费玩家体验和结算稳定 |
| game_controller 可转移 | 房主身份不可转移 | 简化角色模型 |
| 无主游戏房继续开放 | 本局结束后关闭游戏入口，不再展示入口 | 避免用户误以为房主仍在维护游戏房 |

**注意：** 如果 playing 中房主退出或断线，本局可以继续；但本局结束后不能再开新局，房间回到普通语聊房状态。

### 14.3 游戏最小化、房间最小化与麦位兼容

> 本节为结构化规则表格，与第 6 章 6.6-6.11 前台流程互补。6.6-6.11 侧重前台流程与交互细节，本节侧重数据边界、展示规则和兼容矩阵。

游戏座位与语聊房麦位完全独立。**所有已进入游戏容器的用户均可最小化游戏；最小化后只保留游戏入口，不展示麦位；恢复游戏后再同步房间最新麦位、权限、游戏座位、RTC 和钱包状态。**

#### 14.3.1 数据与展示边界

| 类型 | 归属 | 作用 | 数据源 |
|---|---|---|---|
| 房间麦位 Room Mic Seat | 我方语聊房 | 语音聊天、上麦、下麦、锁麦 | RoomMicState |
| 游戏座位 Game Seat | Bytesun / RoomGamePlayer | 玩游戏、观战、结算 | RoomGamePlayer + Bytesun 回调 |
| 游戏内麦位缩略条 | 我方客户端展示层 | 游戏展开态展示谁在麦上、谁在说话 | RoomMicState 镜像 |
| 游戏最小化入口 | 我方客户端展示层 | 恢复游戏入口 | RoomGameSession |
| 当前游戏快捷入口 | 我方客户端展示层 | 用户未进入过游戏时看到当前房间游戏 | RoomGameSession |

麦位展示只存在于两个地方：

1. 房间主界面的完整麦位。
2. 游戏展开态内的麦位缩略条。

游戏最小化入口不展示麦位，避免与房间主界面麦位重复。

#### 14.3.2 角色是否允许最小化

| 角色 / 身份 | 是否可最小化 | 处理 |
|---|---:|---|
| 当前房间房主 | 是 | 收起游戏容器，游戏继续，控制权不变 |
| 普通用户 | 是 | 按普通玩家或观战用户处理 |
| 普通玩家 | 是 | 收起游戏容器，保留游戏座位，游戏继续 |
| 观战用户 | 是 | 收起游戏容器，保留观战态 |
| 未进入游戏的房间用户 | 不属于最小化 | 展示当前游戏快捷入口，可 Join / Watch / Open |
| 被移出游戏座位的用户 | 否 | 不展示恢复到玩家态入口，可按状态展示 Watch / Join |
| 被踢出房间的用户 | 否 | 未参游/观战用户清理入口；playing 玩家默认不允许被踢出游戏 |

#### 14.3.3 最小化入口展示规则

| 信息 | 是否展示 | 说明 |
|---|---|---|
| 游戏名称 | 展示 | 例如 Ludo / UNO |
| 游戏状态 | 展示 | Waiting / Playing / Settling / Finished |
| 游戏人数 | 展示 | 例如 2/4 players |
| 恢复按钮 | 展示 | Open / Continue / Watch |
| 麦位头像 | 不展示 | 避免和房间原生麦位重复 |
| 麦克风状态 | 不展示 | 最小化入口不承载语音状态 |
| 说话动效 | 不展示 | 统一由房间主界面展示 |
| 游戏座位详情 | 不展示 | 点击进入游戏后再展示 |
| 关闭游戏按钮 | 普通用户不展示 | 避免误关；V1 仅当前房间房主在展开态可见 |

入口示例：

```text
Ludo · Playing
2/4 players
[Open]
```

#### 14.3.4 入口状态与点击行为

| 状态 | 浮窗文案 | 点击行为 |
|---|---|---|
| waiting | Waiting | 打开游戏等待页 |
| ready | Ready | 打开准备页 |
| playing | Playing | 恢复游戏容器 |
| watching | Watching | 恢复观战页 |
| your_turn | Your turn | 恢复游戏并高亮当前操作；V1 可不做 |

#### 14.3.5 恢复游戏同步规则

用户点击 Open 恢复游戏时，客户端不得直接使用最小化前缓存。

恢复顺序：

1. 校验用户是否仍在房间。
2. 查询 RoomGameSession，判断游戏是否仍有效。
3. 查询 RoomGamePlayer，判断用户是玩家、观战、未加入、已离开还是被踢。
4. 查询 RoomRoleState，判断用户是否仍为当前房间房主。
5. 查询 RoomMicState，刷新房间最新麦位。
6. 查询 RTCState，刷新静音、禁麦、说话状态。
7. 查询 WalletBalance，刷新游戏币余额。
8. 发送 `gameActionUpdate type=2016 status=expand` 通知 H5 恢复渲染。
9. 等待 500ms H5 恢复渲染。
10. 恢复或重建 H5 游戏容器。
11. 游戏内麦位缩略条按最新 RoomMicState 渲染。

| 恢复场景 | 处理 |
|---|---|
| 游戏仍在 playing | 恢复游戏容器，并按最新 RoomMicState 渲染麦位缩略条 |
| 游戏已 ended | 不恢复旧 H5，提示"当前游戏已结束"，可展示结算结果 |
| 游戏 settling 中 | 恢复结算中页面，不允许重复操作 |
| 用户已被踢出房间 | 未参游/观战用户不允许恢复游戏并清理入口；playing 玩家默认不允许被踢出游戏 |
| 用户游戏座位已被释放 | 恢复为 Watch / Join / Removed 状态 |
| 麦位状态已变化 | 以最新 RoomMicState 为准刷新，不使用最小化前缓存 |
| RoomMicState 拉取失败 | 游戏可先恢复，麦位缩略条展示 loading / retry，不展示旧麦位数据 |
| H5 已被系统回收 | 重新加载 H5，并通过 getConfig 返回最新状态 |

#### 14.3.6 最小化期间兼容场景

| 场景 | 处理 |
|---|---|
| 用户在最小化期间被抱上麦 | 房间主界面立即展示；恢复游戏后麦位缩略条同步为最新状态 |
| 用户在最小化期间被移下麦 | 房间主界面立即更新；恢复游戏后不再展示该用户在麦上 |
| 用户在最小化期间被禁麦 / 禁言 | 恢复游戏后同步禁麦 / 禁言状态；游戏快捷聊天按禁言策略限制 |
| 用户在最小化期间被踢出游戏座位 | 恢复后展示 Removed / Watch / Join，不恢复玩家态 |
| 用户在最小化期间被踢出房间 | 未参游/观战按语聊房逻辑清理；playing 玩家默认不允许被踢出游戏，按 B-08 待确认 |
| 房主在最小化期间离开房间 | waiting / ready 关闭或退款；playing / settling 本局继续，房主游戏角色由系统接管，本局结束后不展示入口 |
| 游戏在最小化期间结束 | 浮窗变为 Finished；点击后展示结果或提示已结束 |
| 游戏在最小化期间进入结算 | 浮窗变为 Settling；点击后展示结算中 |
| 后台强制关闭游戏 | 浮窗隐藏或变为 Closed；点击提示当前游戏已结束 |
| 用户最小化后进入其他房间 | 二次确认；确认后退出原房间和游戏，释放座位并移除浮窗 |
| 用户离开游戏但仍在房间 | 释放游戏座位，不影响房间麦位 |
| 用户离开房间但仍在游戏 | 普通玩家离房同步退出个人游戏态；房主 playing 中离房由系统接管其游戏角色，其他玩家完成本局 |

#### 14.3.7 入口状态刷新机制

> v1.8 新增：最小化入口状态的刷新方式。

| 刷新方式 | 适用场景 | 说明 |
|---|---|---|
| IM 长连接推送 | 游戏状态变更（waiting→playing→settling→ended→closed） | 实时性高，优先采用 |
| 定时轮询（30s） | 推送失败时的兜底 | 客户端定时查询 RoomGameSession.status |
| 用户主动刷新 | 点击入口时触发 | 恢复流程第一步即为查询最新状态 |

**closed 状态触发条件明确定义：**

| 触发源 | 处理 |
|---|---|
| 房主在展开态关闭游戏 | 推送 session_closed → 所有用户入口隐藏 |
| 后台强制关闭 | 推送 session_closed |
| session 自然 ended 后超过 5 分钟无再来一局 | 自动 closed |
| 服务端检测到异常不可恢复 | 推送 session_closed |

#### 14.3.8 房主、普通玩家的交互差异

| 身份 | 最小化后 | 恢复后可见操作 |
|---|---|---|
| 当前房间房主 waiting / ready | 游戏等待态保留 | Start / Invite / Kick Seat / Close |
| 当前房间房主 playing | 本局继续 | 查看游戏、处理异常，关闭按权限和状态控制 |
| 普通玩家 | 保留座位，当前局继续 | 继续游戏；若已被移出座位则展示 Removed / Watch / Join |
| 观战用户 | 保留观战态 | 继续观战；V1 默认 playing 中不支持补位 |
| 未进入游戏用户 | 看到当前游戏快捷入口 | waiting 可 Join，playing 可 Watch / Open |

### 14.5 用户断线、杀 App 与恢复

| 场景 | 处理 |
|---|---|
| 普通用户短暂断线 0-30 秒 | 保留游戏座位，标记 reconnecting |
| 普通用户断线 30-120 秒 | 继续保留座位，房间内标记离线中 |
| 普通用户断线超过 120 秒 | 若游戏支持托管则托管；不支持则标记 left |
| 房主断线 0-30 秒 | 标记 reconnecting，游戏继续 |
| 房主断线 30-120 秒 | 标记离线中，普通用户可见断线提示 |

| 用户杀 App 后重进 | 查询当前房间和 RoomGameSession，恢复游戏 |
| 用户后台返回前台 | 拉取最新游戏状态、座位、麦位和余额 |
| 用户结算后重进 | 展示结算结果或当前游戏已结束提示 |

### 14.11 邀请、分享与活动任务

| 场景 | 处理 |
|---|---|

### 14.13 用户主动退出房间的游戏处理

用户主动退出语聊房时，必须同步处理当前房间内的游戏状态。游戏不可脱离语聊房独立存在。

#### 14.13.1 基础原则

| 原则 | 说明 |
|---|---|
| 退出房间 ≠ 最小化游戏 | 最小化只收起游戏容器；退出房间会退出当前房间关系链 |
| 退出房间必须处理游戏资格 | 若用户是玩家，需释放或标记游戏座位；若是观战，需退出观战 |
| 结算状态不可打断 | settling 中不允许取消订单，用户离开后仍按服务端结果结算 |
| 房主退出按状态处理 | waiting / ready 关闭并退款；playing 中房主退出游戏但不强制中断其他玩家本局，系统接管房主游戏角色 |

#### 14.13.2 按用户身份处理

| 用户身份 | 退出房间时处理 |
|---|---|
| 普通观战用户 | 直接退出观战，移除游戏入口，不影响游戏 session |
| 普通游戏玩家 | 弹窗二次确认；确认后退出房间并退出游戏，座位按状态释放 / left / 托管 |
| 当前房间房主 | 按 14.2 处理：waiting/ready 关闭退款；playing 中房主本人退出游戏，系统接管其游戏角色；settling 等结算完成 |

#### 14.13.3 按游戏状态处理

| 游戏状态 | 用户退出房间后的处理 |
|---|---|
| selecting | 关闭面板，不创建 session |
| loading | 若未预扣门票，取消 session；若已预扣，退还门票 |
| waiting | 释放座位；若已预扣门票，退还 |
| ready | 释放座位；若人数不满足开局条件，状态回退 waiting |
| playing | 标记 player_status=left；是否判负 / 托管 / 继续由三方游戏规则决定 |
| reconnecting | 用户主动退出后不再保留重连资格，标记 left |
| settling | 用户可离开房间，但结算继续完成，结果通过系统消息或下次进入补展示 |
| ended | 正常退出，移除入口 |

#### 14.13.4 交互文案

普通玩家退出房间确认：

```text
你正在游戏中，离开房间将同步退出当前游戏，是否继续？
```

#### 14.13.5 当前房间房主关闭游戏但不退出房间

参考 TopTop 的 `Close Game Mode` 语义，但保持 WeChill 的 RoomGameSession 模型：房主关闭的是当前游戏 session，房间本身继续存在并恢复普通语聊房状态。

```mermaid
flowchart TD
    A["当前房间房主点击关闭游戏"] --> B{"当前游戏状态"}
    B -->|"waiting / ready"| C["确认弹窗：关闭后已加入玩家将被移出"]
    B -->|"playing"| D["确认弹窗：关闭游戏将同步退出当前游戏"]
    B -->|"settling"| E["不允许关闭：结算中请稍候"]
    C --> F{"确认关闭？"}
    D --> F
    F -->|"取消"| G["继续当前游戏"]
    F -->|"确认"| H{"当前状态"}
    H -->|"waiting / ready"| J["关闭 RoomGameSession，退还已预扣门票"]
    H -->|"playing"| K["房主本人走普通用户退出游戏逻辑"]
    K --> L["房主游戏角色由系统接管"]
    L --> M["其他游戏中的用户继续本局"]
    M --> N["本局结束后所有游戏用户退回语聊房"]
    J --> O["清理游戏入口和浮窗"]
    N --> O
    O --> P["房间恢复普通语聊房状态，不展示游戏入口"]
```

| 状态 | 关闭规则 |
|---|---|
| waiting / ready | 允许关闭，退还已预扣门票，释放游戏座位 |
| playing | 二次确认后房主本人退出当前游戏，系统接管房主本局游戏角色；其他玩家继续本局，本局结束后不展示入口 |
| settling | 不允许关闭，必须等待结算完成 |
| ended | 释放入口，房间恢复普通语聊房状态 |

房主关闭游戏提示：

```text
关闭后你将退出当前游戏，其他玩家可完成本局；本局结束后房间恢复为普通语聊房，是否继续？
```

| leave_source | room_exit / switch_room / app_close |

### 14.14 用户被踢出房间的游戏处理

> **v2.0 关键变更：** 游戏中的用户默认不允许被房主通过“踢出房间”影响本局游戏，避免付费结算和对局公平性争议。该项作为 B-08 业务待确认保留；若业务最终要求允许踢出，再按异常退出和结算补偿规则单独评审。

踢人下麦、踢人出房间沿用现有语聊房权限与提示；但处于游戏 playing 中的用户，不允许被房主通过踢出房间直接踢出游戏。

#### 14.14.1 处理规则

| 游戏状态 | 被踢出房间后的处理 |
|---|---|
| 未加入游戏且未观战 | 走原有语聊房踢出房间逻辑，离开房间并清理入口 |
| watching | 离开房间，同步退出观战 |
| waiting / ready 玩家 | 待业务确认；默认可按退出游戏处理并释放座位、退款/解冻 |
| playing 玩家 | 默认不允许踢出游戏；提示房主“该用户正在游戏中，本局结束前不可踢出游戏” |
| reconnecting 玩家 | 按 playing 玩家处理，避免通过断线窗口绕过限制 |
| settling | **不打断结算；订单继续完成，结果通过系统消息或钱包流水可查** |

**核心原则：订单一旦进入 playing / settling / success 关键链路，不因房间管理操作随意回滚。被踢相关策略必须保障付费用户权益和结算可追溯。**

#### 14.14.2 与踢出游戏座位的区别

| 行为 | 影响房间身份 | 影响麦位 | 影响游戏座位 | 是否还能留在房间 |
|---|---:|---:|---:|---:|
| 踢出游戏座位 | 否 | 否 | 是 | 是 |
| 踢人下麦 | 否 | 是 | 否 | 是 |
| 踢出房间（未参游/观战） | 是 | 是 | 观战随房间退出 | 否 |
| 踢出房间（playing 玩家） | 待确认；默认不允许影响本局 | 待确认 | 默认不影响本局 | 默认保留至本局结束 |

#### 14.14.3 用户提示

```text
该用户正在游戏中，本局结束前不可踢出游戏。
```

#### 14.14.4 服务端记录字段

| 字段 | 说明 |
|---|---|
| leave_reason | kicked_from_room |
| operator_user_id | 操作人 |
| room_id | 房间 ID |
| session_id | 游戏 session |
| seat_no | 游戏座位号，可为空 |
| refund_status | none / pending / success / failed / compensating |

### 14.16 用户被踢出游戏座位但仍在房间

### 14.17 用户断线、杀 App、重连恢复详细规则

断线恢复是游戏类场景高频问题，需要区分网络断线、App 后台、杀进程、H5 被系统回收等情况。

#### 14.17.1 断线类型

| 类型 | 判断 | 处理 |
|---|---|---|
| 短断线 | 0-30 秒内心跳恢复 | 保留座位，状态 reconnecting，用户无感恢复 |
| 中断线 | 30-120 秒 | 保留座位，房间内可显示离线中；恢复时同步最新状态 |
| 长断线 | 超过 120 秒 | 标记 left 或进入三方托管，具体按游戏能力 |
| 杀 App | 客户端进程结束，H5 销毁 | 下次启动查询 current_room_id 和 current_game_session_id 重建 |
| H5 被系统回收 | App 仍在房间，WebView 被回收 | 重新加载 H5，通过 getConfig 返回最新状态 |

#### 14.17.2 恢复顺序

```text
查询用户当前房间
→ 查询 RoomGameSession
→ 查询 RoomGamePlayer
→ 查询 RoomRoleState
→ 查询 RoomMicState
→ 查询 RTCState
→ 查询 WalletBalance
→ 发送 gameActionUpdate type=2016 status=expand
→ 判断 H5 恢复 / 重建 / 展示结果 / 提示结束
```

#### 14.17.3 恢复结果

| 查询结果 | 客户端处理 |
|---|---|
| session=playing，player=playing | 恢复游戏 H5 |
| session=playing，player=left | 展示 Watch / 已退出 |
| session=settling | 展示结算中 |
| session=ended，result_viewed=false | 补展示结算结果 |
| session=ended，result_viewed=true | 回到房间，隐藏入口 |
| room closed / kicked | 不恢复游戏，回到房间外页面 |

#### 14.17.4 字段要求

| 字段 | 说明 |
|---|---|
| last_active_at | 用户最后活跃时间 |
| disconnect_at | 断线时间 |
| reconnect_deadline | 重连截止时间 |
| reconnect_count | 当前 session 内重连次数 |
| recover_source | foreground / app_restart / h5_reload |
| recover_result | success / failed / ended / no_permission |

### 14.18 同账号多端登录与并发操作

同一账号不允许在同一游戏 session 中占多个游戏座位，结算必须按 user_id + order_id 幂等处理。

### 14.19 H5、App、服务端状态不一致

RoomGameSession 为游戏状态最终依据，H5 状态仅作为状态推进信号，不作为最终可信状态。

#### 14.19.1 冲突处理

| 冲突 | 处理 |
|---|---|
| H5=playing，服务端=waiting | 校验 type=13；合法则推进 playing，不合法则通知 H5 refresh |
| H5=ended，服务端=playing | 等待 type=14 或服务端结算回调；超时进入异常对账 |
| App=ended，H5=playing | App 通知 H5 refresh / close，不允许继续操作 |
| H5 destroy，服务端=playing | 展示恢复 / 重连，不直接结束 session |
| H5 无响应 | 客户端重试；超过阈值进入 error |
| App 本地缓存与服务端不一致 | 丢弃本地缓存，使用服务端状态重建视图 |

#### 14.19.2 幂等要求

| 事件 | 幂等键 | 处理 |
|---|---|---|
| 游戏开始 | session_id + round_id + type=13 | 已 playing 不重复处理 |
| 游戏结束 | session_id + round_id + type=14 | 已 ended / settling 不重复处理 |
| 扣门票 | ticket_order_id | 不重复扣费 |
| 结算 | order_id | 不重复加减钱 |
| 退款 | refund_order_id | 不重复退款 |

后台需能查看以下日志：

| 日志 | 说明 |
|---|---|
| h5_state_report_log | H5 上报状态日志 |
| app_state_change_log | App 状态切换日志 |
| server_session_state_log | 服务端 session 状态变更 |
| state_conflict_log | 状态冲突记录 |
| idempotent_hit_log | 幂等命中记录 |

### 14.23 游戏中用户身份变化

用户身份变化会影响游戏入口和控制按钮展示，客户端不能只依赖进入游戏时的本地缓存。V1 游戏控制权只归当前房间房主。

| 身份变化 | 处理 |
|---|---|
| 普通用户升为房管 / 管理员 | V1 游戏内仍按普通用户处理，不展示 Start / Close / Kick Seat |
| 房管 / 管理员被取消 | 不影响游戏内控制权，因为 V1 未开放房管控制游戏 |
| 房间归属转让 | 新房间归属用户获得游戏控制权，RoomRoleState 推送后刷新按钮 |
| 用户被封禁 | 退出游戏或挂起结算，按风控策略处理 |

操作权限同步触发点：

| 触发点 | 要求 |
|---|---|
| 打开游戏控制面板 | 重新拉取 RoomRoleState |
| 点击 Start / Close / Kick Seat | 服务端实时校验 |
| 房间归属变化 | 推送权限变更给当前游戏用户 |

### 14.24 游戏中房间上锁、改私密、设置密码

房间权限变化只影响新进入用户，不影响已在房间内且已通过权限校验的用户。

| 场景 | 处理 |
|---|---|
| 已在房间用户 | 不受影响，可继续游戏 / 观战 |
| 新用户进入 | 先校验房间权限，再校验游戏权限 |
| 游戏邀请 | 被邀请人必须先通过房间权限校验 |
| Game Rooms 列表 | 展示锁标识或隐藏，按产品策略配置 |
| 分享链接 | 先校验房间，再校验游戏 session |
| 房间改私密后当前游戏 playing | 本局继续，不允许未授权用户进入 |

### 14.26 Discover Private room 快捷入口边界

Private room 快捷入口是 Discover 侧的快速建房开游链路。入口不同，但底层路径与房内游戏统一：系统进入用户自己的房间，打开游戏设置页，确认配置后创建 RoomGameSession 并进入 H5。它不复用房内 Games 面板的活动和动态表情 Tab，但最小化、恢复、退出和结算均回到自己的房间上下文。

| 场景 | 处理 |
|---|---|
| 用户点击 Private room | 当前 Discover Games 页展示休闲游戏半屏 |
| 半屏内游戏维护 / 下架 | 游戏灰态或隐藏；点击不可用游戏只展示提示 |
| 地区或审核账号限制 | 服务端返回可见游戏列表，客户端不展示受限游戏 |
| 用户当前在他人房间 | 先退出当前他人房间，再进入自己的房间开局 |
| 用户当前在自己的房间 | 若无游戏，打开设置页并在 Start 后创建 RoomGameSession；若已有游戏，提示 Open 当前游戏或关闭后重试 |
| 用户选择游戏 | 关闭半屏，进入用户自己的房间并打开游戏设置页；点击 Start 后预扣门票、创建 RoomGameSession，并进入对应 H5 |
| 首次不展示房间页 | 直接打开游戏设置页，不先展示语聊房主界面 |
| H5 加载失败 | 退还门票，展示 Retry / Close；Close 后回到自己的房间主界面 |
| 游戏币不足 | 预扣失败，打开 Game coins 兑换页 |
| 用户最小化 | 返回自己的房间主界面，并展示当前游戏恢复入口 |
| 用户关闭游戏 | waiting/ready 关闭 RoomGameSession 并回到自己的房间；playing 中按房主退出本局逻辑处理，本局结束后不展示入口 |
| 用户进入其他语聊房 | 若自己房间游戏进行中，二次确认；确认后退出当前游戏和自己的房间 |
| 后台关闭 Private room 入口 | 新用户不展示按钮；已在 H5 的用户本局可完成或按配置关闭 |

埋点与数据要求：

| 字段 / 埋点 | 说明 |
|---|---|
| source=private_room_entry | 标识来源为 Private room 快捷入口 |
| return_context=own_room | 最小化后返回用户自己的房间主界面 |
| private_room_game_picker_show | Private room 休闲游戏半屏曝光 |
| private_room_game_select | 用户在半屏中选择游戏 |

### 14.27 游戏互斥规则

> **v1.8 新增：** 基于 v1.7 的二角色模型重写互斥规则。

#### 14.27.1 核心原则

**同一用户在同一时刻只能在一个房间里。** 因此：
- 要进入别人的游戏房 → 必须先退出当前所在房间
- 要回自己的房间开游戏 → 必须先退出当前所在的别人房间
- 用户只能有一个活跃游戏状态

#### 14.27.2 互斥判断矩阵

| 用户当前状态 | 新操作 | 处理 |
|---|---|---|
| 不在任何房间 | 点击 Game Room / 推荐游戏 | 正常进入（以普通用户） |
| 不在任何房间 | 点击 Private room | 正常进入自己房间开游戏 |
| 在自己房间（有游戏） | 点击 Discover 某个 Game Room | 弹窗确认离开当前房间；确认后按房主退出/当前游戏状态处理，再进入目标 |
| 在自己房间（有游戏） | 再次在 Games 面板选另一个游戏 | 拦截：当前已有游戏进行中 |
| 在别人房间（有游戏） | 点击 Private room | 弹窗确认离开当前房间；确认后退出当前房间和个人游戏态，再进自己房间 |
| 在别人房间（有游戏） | 点击另一个 Game Room | 弹窗确认离开当前房间；确认后退出当前房间和个人游戏态，再进入新 Game Room |
| 在别人房间（无游戏） | 同上 | 弹窗确认离开当前房间；确认后退出当前房间 |
| 有游戏 settling | 任何新操作 | 拦截：结算中，请稍候 |

#### 14.27.3 互斥判断流程

```mermaid
flowchart TD
    A["用户触发新入口"] --> B{"用户当前是否在某个房间"}
    B -->|"不在"| C["直接执行目标操作"]
    B -->|"在某个房间（最小化）"| D{"是否有活跃游戏且状态=settling"}
    D -->|"settling"| E["拦截：结算中，请稍候"]
    D -->|"非 settling 或无游戏"| F["弹窗：确认离开当前房间么？"]
    F -->|"取消"| G["留在当前状态"]
    F -->|"确认"| H["退出当前房间"]
    H --> I{"退出时的游戏状态"}
    I -->|"无游戏"| J["直接退出"]
    I -->|"waiting/ready"| K["释放座位，门票退款"]
    I -->|"playing"| L["标记 left，按三方规则处理"]
    J --> C
    K --> C
    L --> C
```

#### 14.27.4 提示文案

| 场景 | 文案 |
|---|---|
| 在自己房间有游戏，要去别人游戏房 | 确认离开当前房间么？ |
| 在别人房间有游戏，要回自己房间 | 确认离开当前房间么？ |
| 在别人房间有游戏，要去另一个游戏房 | 确认离开当前房间么？ |
| 在别人房间无游戏，要去另一个房间 | 确认离开当前房间么？ |

| 场景 | 处理 |
|---|---|

### 14.33 游戏安全区适配规则

> v1.8 新增：基于 Bytesun 对接文档 v1.0.7 Section 10.1。

**核心原则：** 全屏语聊房模式的游戏画面及内容，图层在 App 麦位等 ICON 下方。由 App 提供安全操作区域。

**URL 参数配置：**

| 参数 | 类型 | 说明 | 默认 |
|---|---|---|---|
| game_margin_top | float | 顶部覆盖区域占全部游戏区的百分比 | 0 |
| game_margin_bottom | float | 底部覆盖区域占全部游戏区的百分比 | 0 |
| game_margin_standard | int | 安全区高度不足时的对齐基准：0=底部对齐，1=顶部对齐 | 0 |

**WeChill 适配策略：**

```mermaid
flowchart TD
    A["加载游戏 H5 URL"] --> B{"game_id 是否以 3 开头"}
    B -->|"是"| C["计算安全区参数"]
    B -->|"否（以 2 开头等）"| D["不添加安全区参数"]
    C --> E["game_margin_top = 顶部状态栏+麦位区域高度 / 屏幕高度"]
    C --> F["game_margin_bottom = 底部操作栏高度 / 屏幕高度"]
    C --> G["game_margin_standard = 0（底部对齐）"]
    E --> H["拼接到 download_url 后面"]
    F --> H
    G --> H
    H --> I["加载完整 URL"]
```

**机型适配注意事项：**

| 机型特征 | 处理 |
|---|---|
| 刘海屏 | game_margin_top 需加上刘海高度 |
| 底部横条（iPhone X+） | game_margin_bottom 需加上安全区高度 |
| 折叠屏展开态 | 重新计算安全区参数，通过 type=2016 通知 H5 刷新 |
| 横屏游戏（game_orientation=2） | 需旋转安全区计算逻辑，left/right 替代 top/bottom |

## 17. 验收标准

### 17.1 产品验收

- 用户可以从 Discover Games 看到推荐游戏、国家筛选和正在玩游戏的语聊房。
- 用户从 Discover 推荐游戏或 Game Rooms 进入时，若当前已有最小化房间，需先退出当前房间，再进入匹配到的游戏房，并在目标房间内按普通用户处理。
- 用户点击 Discover Games 的 `Private room` 后，当前首页弹出休闲游戏选择半屏，弹窗不展示 `游戏`、`动态表情` 两个 Tab。
- 用户在 Private room 半屏中选择游戏后，系统进入用户自己的房间并打开对应游戏设置页；首次不展示语聊房主界面。
- Private room 创建后进入 waiting 状态；房主可最小化等待页、分享邀请链接；有人进入并点击 Join 后，房主侧弹起配置页，可确认是否开局。
- Private room 来源游戏最小化后，返回自己的房间主界面，恢复逻辑与房内游戏一致，不回到 Discover Games 首页。
- Activity 页只展示搜索入口，不展示任务和头像占位。
- 用户可以从房间内 Games 入口打开半屏游戏面板。
- 当前房间房主可以选择游戏并打开设置页。
- 当前房间房主可以开始 Ludo / UNO，其他首批游戏可按灰度配置进入。
- 普通用户可以加入、观战、最小化和返回房间。
- 游戏开始和结束状态能同步到 App。
- 用户可以完成一局并看到结算结果。
- 结算结果页能展示胜负 / 排名 / 游戏币变化 / 退款或奖励摘要，并支持返回倒计时。
- 游戏币不足时可以打开 Game coins 兑换页，金币兑换成功后刷新游戏币余额。
- 最小化后可通过房间浮窗恢复游戏。
- Game Rooms / 房间卡片能展示游戏 Badge、状态和人数信息。
- 房主创建游戏、游戏开始、游戏结束时可按频控发送房间系统消息。
- H5 加载中能展示 Loading 进度文案、Tips、Retry / Close 失败兜底。
- 加载失败、上座失败、维护、封禁、地区限制均有明确提示。
- Private room 入口在 Discover 当前页弹出休闲游戏选择半屏；弹窗只保留休闲游戏，选中游戏后进入自己的房间并打开设置页，首次不展示房间主界面。
- Activity 页活动 Banner 可跳转活动详情、指定房间、指定游戏或金币兑换游戏币页面；活动绑定游戏下架后活动自动灰态。
- 全角色（当前房间房主、普通玩家、观战）均可最小化游戏并通过浮窗恢复。
- 被踢出游戏座位的用户仍可留在房间观战；游戏中的用户默认不可被房主踢出游戏，是否允许踢出房间作为 B-08 待确认。
- 不同角色（房主/普通用户）从不同入口（房内/Discover/Private room/Game Rooms）均可按权限正确进入游戏。
- 用户同时只能有一个活跃游戏，跨来源开新游戏时展示二次确认。
- 门票在 Start/Join 时预扣（frozen），gameLoaded 失败或 type=13 超时时自动退还（refunded）。
- 最小化入口状态能实时更新（推送/轮询）。
- 房主主动退出按游戏状态处理：waiting/ready 关闭并退款；playing 中房主本人退出游戏，房主游戏角色由系统接管，其他玩家完成本局，本局结束后不展示入口。
- 房主断线 > 120s 后按分层处理规则关闭或允许本局继续后关闭。
- playing 中不可通过 UI 踢出玩家座位。
- 若业务最终允许踢出已进入 settling 的用户，订单仍继续完成，用户可通过系统消息或钱包流水查看结果。
- 推荐游戏入口点击后展示 Matching 15s；匹配到同类型 waiting/ready 未开局房直接进入，无房展示空结果。
- 游戏内音频播放正常（背景音乐/音效），与语聊房 RTC 不冲突。
- 最小化时正确发送 type=2016 minimize 通知，恢复时发送 expand。
- 麦克风/扬声器操作在游戏 H5 和语聊房原生 UI 之间双向同步。
- language 跟随 App 设置正确映射。

- game_start 和 game_settle 上报数据完整且 game_round_id 正确关联。
- 游戏开始和结算上报可落库。
- 安全区参数不会遮挡主要游戏操作区。
- 游戏包版本可后台控制。
- 恢复游戏时必须重新拉取 RoomMicState、RoomGamePlayer、RoomRoleState、WalletBalance，不得使用本地缓存。
- 幂等处理覆盖游戏开始、结束、扣门票、结算、退款全链路。
- active_game_session_id 互斥查询在所有入口流程中生效。
- 最小化入口通过 IM 推送实时更新，30s 轮询兜底。
- 签名生成正确（md5，15s 有效期）。
- 安全区参数适配主流机型。
- URL 参数（hideLobby/antiAddictionImg/language）根据配置正确拼接。
- game_id 以 3 开头的游戏正确配置安全区参数。
- Token 过期时自动刷新，不中断游戏。
- 错误码 1008 正确触发充值弹窗。
- 错误码 1019/1020 正确阻止游戏进入。
- game_room_badge_show、game_room_system_message_show / click、game_h5_loading_slow、game_result_countdown_finish 等新增埋点可上报。

## 18. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Bytesun 服务不稳定 | 游戏无法加载或结算失败 | URL / Zip 双方案、重试、下架开关、监控告警 |
| 游戏包加载慢 | 进入转化下降 | CDN 预热、本地包、Loading 进度、首屏优化 |
| 结算并发 | 错账 | 用户级锁、订单幂等、补偿队列、对账 |
| 门票预扣并发 | 预扣与退还竞争 | ticket_order 幂等、用户级锁、超时自动退还 |
| 金币兑换游戏币错账 | 金币扣减或游戏币增加不一致 | 同事务、兑换订单幂等、兑换对账、补偿队列 |
| 语音和游戏声音冲突 | 体验差 | 语音优先，默认降低 BGM，用户可控制，type=3003 音频播放协议 |
| 低端机性能 | 卡顿 / 崩溃 | 机型监控、降级策略，必要时屏蔽高负载游戏 |
| 游戏操作区被遮挡 | 无法操作 | game_margin_top / bottom、安全区验收 |
| 玩法合规 | 审核风险 | 审核账号隐藏、免费局策略、Slots 单独评审 |
| 作弊串通 | 破坏经济系统 | 同设备/IP 风控、异常胜率、固定组合检测 |
| 双 RTC 冲突 | 游戏内 RTC 与语聊房 RTC 冲突 | V1 不接入需要 isGameRTC 的游戏 |

## 20. 评审决策项

| 决策项 | 建议结论 |
|---|---|
| V1 路径 | 采用"语聊房内直接调起游戏"，不新增游戏房类型 |
| 首批游戏 | 6 款进入产品范围，Ludo / UNO 优先强验收，其余按灰度节奏上线 |
| 普通语聊房开启游戏 | 默认支持，按地区/端/当前房间房主等级可后台限制 |
| 门票系统 | V1 启用，采用预扣（frozen）+确认（success）方案，单位为游戏币，默认 0，范围后台配置 |

| 角色模型 | V1 采用房主+普通用户二角色体系，观战为普通用户状态 |
| 房主退出 | V1 不做房间管理员接管；waiting/ready 关闭并退款；playing 中系统仅接管房主本局游戏角色，其他玩家完成本局 |

## 21. 最终推荐

本期采用"现有语聊房内直接调起三方休闲游戏"的方案。

这样可以在不改造房间模型、麦位和基础权限的情况下，最快验证游戏对房间停留、活跃和付费转化的提升。V1 重点跑通 Discover Games、国家筛选、推荐游戏 15s 匹配、Private room 选游后设置页开局、房间内入口、H5 容器、座位同步、门票预扣与确认、游戏币结算、金币兑换游戏币、最小化恢复和后台监控。待核心数据稳定后，再考虑 Game Rooms 深度运营、独立 Quick Match、游戏房类型、任务和锦标赛等扩展能力。

**一句话总结：第一版先把三方休闲游戏做成语聊房里的"可开局插件"，让用户在原房间关系链中一起玩；等数据验证后，再升级成完整游戏房生态。**

## 22. 竞品分析与 WeChill 可借鉴优化

> 本章节基于《WeChill-角色入口场景流程图汇总》中的 TopTop / Hawa 竞品路径整理，单独放在文档最后。竞品仅作为样式与体验参考，不改变 WeChill V1 的核心方案：**语聊房内调起、Discover 半屏选游、Private room 进入自己房间并打开设置页、全屏 H5 WebView、RoomGameSession 单游戏会话**。

### 22.1 竞品对比概览

| 维度 | TopTop | Hawa | WeChill V1 裁决 |
|---|---|---|---|
| 游戏承载 | 原生游戏包 + 房内浮动卡片 | 房间 UI 改造成游戏模式 | 不采用；WeChill 保持全屏 H5 WebView |
| 首页入口 | 游戏网格 → 游戏详情页 → Start / Create / Search / Recommended Rooms | Discover 游戏卡片 → Matching 倒计时自动匹配 | V1 不新增游戏详情页；推荐游戏采纳 15s 匹配等待，Game Rooms 卡片仍直接进入 |
| Private room | 首页 Private room + 创建私密房间开局 | 自己房间 Mini Game 面板开局 | WeChill 先在 Discover 半屏选游戏，再进入自己房间打开设置页 |
| 房内开局 | 房主选择游戏 → Mode / Bet → Confirm | 房主选择游戏 → 人数 / Match / Auto-next → Create | 可参考一屏配置，但 V1 不加入自动匹配和自动下一局 |
| 退出机制 | `Exit Game` 与 `Close Game Mode` 分离 | End game session 确认弹窗 | 采纳语义：普通用户退出游戏不退出房间；房主可关闭当前游戏 session |
| 最小化召回 | Keep / Leave + 首页状态标签 | 顶部 Banner / 覆盖层持续召回 | 参考文案和状态标签；WeChill 仍用浮窗和所属房间恢复 |
| 加载体验 | 下载进度 | Loading 百分比 | 采纳 Loading 进度 / Tips / Retry 兜底 |
| 结果循环 | 未重点体现 | WIN/OUT 结果卡 + 7s Return + Auto-next | 采纳结果卡和返回倒计时；V1 默认不自动下一局 |

### 22.2 TopTop 可参考点

TopTop 的技术形态与 WeChill 不同：TopTop 偏原生包和房内浮动卡片，WeChill 是全屏 H5。可参考的是路径和语义，而不是 UI 容器形态。

| 参考点 | 价值 | WeChill 处理 |
|---|---|---|
| `Exit Game` / `Close Game Mode` 分离 | 避免“个人退出”和“房主关闭游戏”混淆 | 已纳入 14.15 和 14.13.5 |
| Mode + Bet 一屏确认 | 降低房主开局配置负担 | 纳入设置页体验优化；V1 保留后台配置项 |
| 首页状态召回文案 | 提升最小化后回流 | 可用于游戏浮窗文案，如 “Ludo · Waiting for players” |
| Recommended Rooms | 提升游戏房发现效率 | V1 已有 Game Rooms；V2 可评估游戏详情聚合页 |
| 浮动游戏卡片 | 与 WeChill 全屏 H5 冲突 | 不采用 |

TopTop 退出语义映射：

```mermaid
flowchart TD

  B -->|"Exit Game"| C["仅退出个人游戏座位"]
```
