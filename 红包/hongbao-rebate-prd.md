# 红包玩法概率返奖 PRD

## 1. 文档信息

- 文档名称：红包玩法概率返奖 PRD
- 适用产品：中东语聊房 / 语音房红包玩法
- 输出目标：在现有红包玩法基础上，新增“按红包总金额达门槛后参与概率返奖”的机制

---

## 2. 需求背景

现有红包玩法已经具备以下基础能力：

1. 房间内发红包入口
2. 金币红包 / 礼物红包两种形式
3. 红包领取条件
   - 在麦用户
   - 在房用户
4. 红包发放方式
   - 随机
   - 均分
5. 发红包动画、公屏消息、全服飘屏
6. 红包过期未领完时，剩余金额返还发送人

现阶段红包玩法主要承担：

1. 拉动房间内互动氛围
2. 提升用户送金币 / 消费意愿
3. 提高房间停留与抢红包参与度

但现有玩法仍缺少一个“发送端激励点”：

1. 用户发出大额红包后，除了房间氛围收益，没有额外的运气型刺激
2. 大额红包与小额红包的发送体验差异不够明显
3. 红包玩法缺少“可反复尝试”的概率激励闭环

因此需要在现有玩法上新增：

`红包总金额达标后，可参与概率返奖；中奖后返还红包总金额的一定比例。`

---

## 3. 需求目标

### 3.1 业务目标

1. 刺激用户发送更高总金额的红包
2. 拉升红包玩法的总 GMV
3. 提高红包被快速领完的比例
4. 强化房间内“发红包 - 抢红包 - 返奖”的闭环刺激

### 3.2 体验目标

1. 用户能清楚理解返奖资格
2. 用户能直观看到“差多少达标”
3. 用户能在发红包后感知到开奖期待
4. 中奖反馈明确，未中奖不打断主流程

### 3.3 成本目标

概率返奖本质属于平台补贴，需将补贴控制在合理范围内。

本次推荐模型的目标期望补贴率：

`0.5% GMV`

即：

`返奖概率 × 返奖比例 = 期望补贴率`

## 4. 现有玩法梳理（基于原 PDF）

### 4.1 基础规则

1. 已上锁房间不可发送红包
2. 发送红包入口有展示门槛：财富等级 10 级以上用户才可见
3. 红包分为：
   - 金币红包
   - 礼物红包
4. 红包个数不可为 1

### 4.2 金币红包

支持配置：

1. 总金额
2. 可领取份数
3. 领取条件
   - 在麦用户
   - 在房用户
4. 发放类型
   - 随机
   - 均分

### 4.3 礼物红包

支持配置：

1. 礼物选项
2. 礼物数量
3. 总金额 = 礼物单价 × 礼物数量
4. 可领取份数
5. 领取条件

### 4.4 发送与领取

1. 发送前校验金币余额
2. 发送成功后触发：
   - 发红包动画
   - 红包横幅
   - 公屏消息
   - 达到高额条件时可触发全服飘屏
3. 用户点击红包悬浮窗后打开领取弹窗
4. 红包被领完 / 过期后，展示不同结果弹窗
5. 未领取完部分会返还给发送人

## 5. 新增玩法概述

### 5.1 玩法定义

当红包总金额达到返奖门槛时，该红包获得“返奖资格”。

若该红包满足有效结算条件，则在结算时触发一次概率抽奖：

1. 按固定概率判断是否中奖
2. 若中奖，则返还红包总金额的固定比例
3. 返奖金额返还给红包发送人

### 5.2 核心原则

1. 返奖只奖励发送人，不奖励领取人
2. 返奖资格按“红包总金额”判断
3. 返奖金额按“红包总金额 × 固定返奖比例”计算
4. 概率与返奖比例固定，不做阶梯倍率
5. 为避免薅羊毛，返奖不在发送成功时立即结算，而在红包有效结算时开奖

## 6. 概率返奖模型推荐

### 6.1 推荐模型

建议首版默认配置如下：

- 返奖门槛：`红包总金额 >= 10,000 金币`
- 返奖概率：`5%`
- 返奖比例：`红包总金额的 10%`

即：

当一个红包总金额达到 10,000 金币及以上，且满足返奖结算条件时：

- 有 `5%` 的概率中奖
- 中奖后返还 `红包总金额 × 10%`

### 6.2 模型公式

#### 返奖资格判断

`红包总金额 = 单个红包金额 × 红包个数`

或对于礼物红包：

`红包总金额 = 礼物单价 × 礼物数量`

若：

`红包总金额 >= 10,000`

则该红包具备返奖资格。

#### 返奖金额计算

`返奖金额 = 红包总金额 × 10%`

结果取整规则建议：

- 向下取整到整数金币

#### 期望补贴率

`5% × 10% = 0.5%`

即平台长期期望补贴约等于：

`合格红包 GMV 的 0.5%`

### 6.3 选择该模型的原因

#### 从用户感知看

1. 10,000 金币是一个足够清晰的大额门槛，便于理解
2. 5% 概率不会过高，仍有“开奖刺激感”
3. 10% 返奖金额用户能明显感知中奖收益
4. 门槛、概率、返奖比例都很直观，规则简单

#### 从成本控制看

1. 0.5% 的期望补贴率较为可控
2. 不会对整体红包 GMV 造成过大补贴压力
3. 适合作为首版上线模型

#### 从增长效果看

1. 可以明显推动用户把红包总额往门槛附近抬高
2. 有利于提升“大额红包占比”
3. 有利于提升红包被尽快领完的房间氛围

### 6.4 示例

| 红包总金额 | 是否达标 | 中奖概率 | 返奖比例 | 中奖返奖金额 | 期望返奖成本 |
| --- | --- | --- | --- | --- | --- |
| 5,000 | 否 | 0 | 0 | 0 | 0 |
| 10,000 | 是 | 5% | 10% | 1,000 | 50 |
| 20,000 | 是 | 5% | 10% | 2,000 | 100 |
| 50,000 | 是 | 5% | 10% | 5,000 | 250 |
| 100,000 | 是 | 5% | 10% | 10,000 | 500 |

## 7. 返奖资格与结算规则

### 7.1 返奖资格

红包满足以下条件后，展示“可参与返奖”标识：

1. 红包总金额 >= 10,000 金币
2. 红包个数 > 1
3. 红包发送成功

### 7.2 有效结算条件

这是本需求的关键规则，用于避免玩法被薅。

建议只有满足以下条件的红包，才真正触发返奖抽奖：

1. 红包总金额 >= 10,000 金币
2. 红包在有效期内被 `全部领完`
3. 有效领取人数 >= 2
4. 发送人本人领取，不计入“有效领取人数”

### 7.3 为什么要求“领完后开奖”

原玩法中：

1. 红包未领完部分会返还给发送人
2. 发送人自己也可以领取红包

如果红包只要发出就参与返奖，会产生明显可薅空间：

1. 用户可发送高额红包但不让别人领完
2. 到期后拿回剩余金额
3. 同时仍保留返奖抽奖资格

因此建议：

`只有红包被领完后，才触发返奖抽奖`

这样可以同时实现两件事：

1. 避免用户低风险薅返奖
2. 反向激励发送人推动房间用户积极抢完红包

### 7.4 不满足结算条件时

如果红包满足返奖门槛，但最终：

1. 未领完
2. 有效领取人数不足 2
3. 红包过期

则：

1. 仍按原逻辑返还未领取部分
2. 不参与返奖抽奖
3. 不展示“未中奖”提示，只展示“未满足返奖结算条件”

## 8. 用户流程

### 8.1 发送红包流程

1. 用户打开礼物面板，点击红包入口
2. 选择金币红包或礼物红包
3. 选择：
   - 总金额 / 礼物数量
   - 领取份数
   - 领取条件
   - 发放类型
4. 系统实时计算红包总金额
5. 系统根据总金额判断是否达到返奖门槛
6. 用户点击发送
7. 校验余额成功后，发送红包并进入房间展示

### 8.2 返奖判断流程

1. 红包发送成功后，若总金额达标，则该红包标记为“返奖候选红包”
2. 红包进入正常领取流程
3. 若红包在有效期内被领完，进入返奖抽奖
4. 系统按 5% 概率判断是否中奖
5. 若中奖：
   - 返还红包总金额 10% 到发送人钱包
   - 展示中奖结果
6. 若未中奖：
   - 不额外打断主链路
   - 可在记录页查看“已参与未中奖”

### 8.3 红包过期流程

1. 红包到期
2. 未领取部分按原逻辑返还发送人
3. 若红包未领完，则不参与返奖抽奖

## 9.1 发红包弹窗改动

在发红包弹窗底部新增“返奖提示区”。

#### 未达标时

展示文案：

`再发 XX 金币，即可解锁返奖资格`

`再发 2,000 金币，即可解锁 5% 概率返 10% 的惊喜返奖`

#### 达标时

展示文案：

`该红包已解锁返奖资格：领完后有 5% 概率返还总金额的 10%`

### 展示优先级

1. 规则文案不应遮挡核心配置项
2. 建议放在发送按钮上方
3. 达标态使用高亮提示色

## 9.2 玩法说明页改动

在原规则说明页新增“返奖规则”模块。

建议文案：

1. 当红包总金额达到 10,000 金币及以上时，可参与返奖
2. 只有红包被领完后才会开奖
3. 中奖概率为 5%
4. 中奖后返还红包总金额的 10%
5. 未领完红包不参与返奖

对于满足返奖资格的红包，在红包悬浮窗或红包横幅处增加轻提示：

`该红包领完后可开奖`

目的：

1. 提示房间用户尽快领取
2. 放大房间参与感

## 9.4 发送人结果反馈

### 中奖

在红包领完并抽奖命中后，发送人收到结果弹窗：

标题：

`恭喜获得返奖`

内容：

`你的红包已被抢完，成功触发返奖，{返奖金额}金币已返还至钱包`

### 未中奖

建议不立即弹强打扰弹窗。

处理方式：

1. 弱提示 toast：
   - `红包已领完，已参与返奖抽奖`
2. 或仅在记录页展示结果

### 未满足开奖条件

若红包过期未领完，则发送人结果页展示：

`红包未领完，剩余金额已返还，本次未触发返奖抽奖`

1. 单用户每日最大中奖次数
2. 单用户每日最大返奖总额

这样可以控制极端情况下的补贴波动。

## 11. 服务端结算逻辑

### 11.1 关键字段

每个红包新增以下字段：

- `total_amount`
- `is_rebate_eligible`
- `rebate_threshold`
- `rebate_probability`
- `rebate_ratio`
- `is_rebate_settled`
- `rebate_status`
  - not_eligible
  - waiting_settle
  - not_triggered
  - lost
  - won
- `rebate_amount`
- `qualified_receiver_count`
- `is_fully_claimed`

### 11.2 服务端结算流程

1. 红包创建成功
2. 判断是否达到返奖门槛
3. 若未达标：
   - `rebate_status = not_eligible`
4. 若达标：
   - `rebate_status = waiting_settle`
5. 每次有人领取红包时，刷新：
   - 已领取份数
   - 有效领取人数
   - 是否领完
6. 当红包被领完时：
   - 判断是否满足最低有效领取人数
   - 若满足，触发抽奖
7. 随机命中：
   - `中奖：rebate_status = won`
   - `未中奖：rebate_status = lost`
8. 若红包过期未领完：
   - `rebate_status = not_triggered`

### 11.3 随机实现建议

采用服务端统一随机，不允许客户端本地判断。

建议：

1. 结算时按唯一红包 ID 做一次抽奖
2. 抽奖结果写入日志
3. 保证幂等
4. 若结算重试，不重复开奖

## 12. 返奖发放规则

### 12.1 奖励发放对象

红包发送人

### 12.2 奖励类型

金币

### 12.3 发放时间

建议在红包领完并完成抽奖判断后立即发放。

### 12.4 金额计算

`返奖金额 = 红包总金额 × 10%`

示例：

- 红包总金额 10,000 -> 返奖 1,000
- 红包总金额 23,000 -> 返奖 2,300

### 12.5 发放失败

若返奖失败：

1. 自动重试 3 次
2. 重试失败写入补发队列
3. 记录发奖状态

## 13. 风险控制

这是本方案的重点，必须与返奖逻辑一同上线。

### 13.1 风险点

1. 用户自己发自己抢，套取返奖
2. 同设备多账号互抢红包，制造“领完”条件
3. 低成本刷返奖概率
4. 同一用户高频发大额红包，集中薅补贴

### 13.2 风控策略

#### 规则一：发送人自领不计入有效领取人数

目的：

避免用户通过自领凑人数。

#### 规则二：红包需领完才开奖

目的：

避免用户只发不领、到期退回仍参与开奖。

#### 规则三：最低有效领取人数 >= 2

目的：

保证红包确实完成基本社交分发。

#### 规则四：设备/IP/账号聚集识别

当命中以下情况时，红包不参与返奖：

1. 有效领取用户中，多个账号同设备
2. 有效领取用户中，多个账号同高风险 IP
3. 高频互抢关系明显异常

#### 规则五：日返奖上限

目的：

限制大 R 高频连续命中带来的补贴波动。

### 13.3 风控命中处理

1. 不参与返奖抽奖
2. 正常保留红包原有领取与退回逻辑
3. 记录风控命中日志
4. 高风险账户进入审核队列

## 16. 边界场景

### 16.1 红包发送成功但未达门槛

按原红包逻辑执行，不参与返奖。

### 16.2 达门槛但红包过期未领完

1. 未领取部分返还发送人
2. 不开奖

### 16.3 达门槛且红包领完但用户未中奖

1. 红包正常结束
2. 不返奖

### 16.4 达门槛且红包领完且用户中奖

返奖立即发放到发送人钱包。

### 16.5 礼物红包是否可参与返奖

建议可以参与。

原因：

1. 礼物红包也有明确总金额
2. 可统一用户认知
3. 有利于礼物红包发送提升

### 16.6 发送人自己领取红包

允许，但不计入有效领取人数。

## 17. 推荐上线策略

### 17.1 首版建议

采用单一固定模型：

- 门槛：10,000 金币
- 概率：5%
- 比例：10%

不建议首版上线：

1. 多档位返奖概率
2. 多档位返奖比例
3. 随机倍率返奖
4. 复杂大奖池

原因：

1. 规则更简单
2. 研发成本更低
3. 用户更容易理解
4. 更便于测算真实 ROI

### 17.2 灰度建议

建议按以下节奏上线：

1. 小流量灰度 5%
2. 核验：
   - 红包 GMV 是否上升
   - 大额红包占比是否上升
   - 实际补贴率是否稳定在预期附近
   - 是否有明显作弊样本
3. 稳定后扩大到 20%、50%、100%

## 18. 最终方案结论

本次推荐的红包返奖模型为：

### 默认返奖模型

- 返奖门槛：`红包总金额 >= 10,000 金币`
- 返奖概率：`5%`
- 返奖比例：`10%`
- 返奖对象：`红包发送人`
- 返奖时间：`红包领完后立即开奖并返还`

### 核心防薅规则

- 红包必须 `领完` 才开奖
- 有效领取人数至少 `2` 人
- 发送人自领 `不计入` 有效领取人数
- 风控命中红包 `不参与返奖`
- 支持 `每日中奖次数 / 金额上限`

### 方案优点

1. 对用户规则简单易懂
2. 对发送大额红包有明确刺激
3. 期望补贴率 0.5%，成本相对可控
4. 与现有红包玩法兼容度高
5. 能有效带动“尽快抢完红包”的房间互动氛围
