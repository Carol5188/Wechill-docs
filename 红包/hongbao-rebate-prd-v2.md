## 2. 本版 PRD 关键变化

基于新版 PDF，这次和上一版相比有 4 个关键变化：

1. 红包领取倒计时明确为 `3 分钟`
2. 红包返奖门槛明确改为 `20k / 50k` 两档
3. 已明确：

因此本版不再沿用单一 `10k/5%/10%` 模型，而改为按新版文档采用两档返奖。

## 4. 概率返奖模型

## 4.1 返奖分档

### 金币红包

| 红包总金额 | 是否参与返奖 | 返奖概率 | 返奖比例 |
| --- | --- | --- | --- |
| 3,000 | 否 | 0 | 0 |
| 5,000 | 否 | 0 | 0 |
| 10,000 | 否 | 0 | 0 |
| 20,000 | 是 | 3% | 5% |
| 50,000 | 是 | 5% | 10% |

### 礼物红包

| 礼物总价值 | 是否参与返奖 | 返奖概率 | 返奖比例 |
| --- | --- | --- | --- |
| < 20,000 | 否 | 0 | 0 |
| 20,000 - 49,999 | 是 | 3% | 5% |
| >= 50,000 | 是 | 5% | 10% |

### 4.2 返奖公式

#### 返奖资格判断

金币红包：

`红包总金额 = 选中总金额`

礼物红包：

`红包总金额 = 礼物单价 × 礼物数量`

#### 返奖金额计算

`返奖金额 = 红包总金额 × 返奖比例`

金币值向下取整。

### 4.3 示例

| 红包类型 | 总金额 | 档位 | 概率 | 比例 | 返奖金额 |
| --- | --- | --- | --- | --- | --- |
| 金币红包 | 20,000 | A档 | 3% | 5% | 1,000 |
| 金币红包 | 50,000 | B档 | 5% | 10% | 5,000 |
| 礼物红包 | 26,640 | A档 | 3% | 5% | 1,332 |
| 礼物红包 | 66,640 | B档 | 5% | 10% | 6,664 |

### 4.4 期望补贴率

#### A档

`3% × 5% = 0.15%`

#### B档

`5% × 10% = 0.50%`

结论：

1. A 档适合作为“中高金额红包”刺激
2. B 档适合作为“房间秀场红包”刺激
3. 整体补贴可控，且梯度清晰

## 5. 返奖触发规则

### 5.1 返奖资格

红包满足以下条件时，具备返奖资格：

1. 红包总金额满足对应档位门槛
2. 红包个数 > 1
3. 发送成功

### 5.2 有效开奖条件

红包只有在以下条件全部满足时，才触发返奖抽奖：

1. 红包达到返奖门槛
2. 红包在 3 分钟倒计时内被领完
3. 有效领取人数 >= 2
4. 发送人本人领取不计入有效领取人数
5. 发送人未处于限制级或危险级红包黑名单
6. 未命中风控规则
7. 未触发平台日返奖预算熔断

### 5.3 返奖结算时机

`红包领完后立即开奖并返还`

### 5.4 不触发返奖的场景

以下场景不触发返奖：

1. 红包未领完
2. 有效领取人数不足 2
3. 命中风控拦截
4. 发送人处于限制级或危险级红包黑名单
5. 已达到平台单日返奖预算上限
6. 发奖失败且未补发成功

## 6. 风控设计补充

这是新版最需要补充完整的部分。本期风控不仅拦截单个红包，还需要支持红包黑名单、关联设备 / IP 同步限制、飞书预警、预算熔断、发奖幂等、生产环境模拟器限制和防脚本抢红包。

## 6.1 风控目标

1. 避免用户通过自领或小号互抢套取返奖
2. 避免通过“发出后不领完”低风险薅返奖
3. 控制大额红包连续命中导致的补贴波动
4. 识别高风险房间和固定团伙互抢
5. 防止脚本毫秒级监听 IM 消息并自动调用领取接口
6. 保持正常房间红包互动体验不被误伤

## 6.2 红包黑名单

### 6.2.1 黑名单来源

1. 命中风控规则的账号自动进入风险名单
2. 命中账号的关联设备 / 关联高风险 IP 下所有账号同步限制返奖资格
3. 运营巡房发现异常房间、固定互抢团伙时，可手动添加风险用户
4. 用户投诉后经客服或风控运营确认，可手动添加风险用户

### 6.2.2 黑名单等级

| 等级 | 入口展示 | 发红包 | 领红包 | 返奖资格 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 危险级 | 不展示红包入口 | 禁止 | 禁止 | 无 | 明确作弊、脚本抢红包、模拟器生产套利、团伙账号主号 |
| 限制级 | 展示 | 允许 | 允许 | 无 | 关联设备 / IP 账号、高风险房间发送人、异常高频大额用户 |

说明：

1. 危险级账号不可看到红包入口，服务端同时拒绝发红包与领红包接口
2. 限制级账号可正常发红包、领红包，但其发出的红包不参与高额红包概率返奖
3. 自动关联限制默认进入限制级；如有直接作弊证据，再升级为危险级
4. 黑名单变更必须记录操作人、来源、原因、有效期、关联设备 / IP 快照

## 6.3 风控规则

### 规则 1：同设备 / 同 IP 多账号互抢识别

命中条件建议：

1. 同一红包的有效领取用户中，有多个账号来自同设备
2. 同一红包的有效领取用户中，有多个账号来自高风险同 IP
3. 同一发送人与固定一组账号高频互抢

命中处理：

1. 该红包不参与返奖
2. 记录风控日志
3. 进入风控审核队列
4. 命中账号及关联设备 / IP 下账号同步进入限制级黑名单；证据明确时升级危险级

### 规则 2：异常高频大额红包识别

该规则只做红包风险预警和限制级黑名单处理，不直接影响普通小额红包体验。预警通过飞书机器人发送到专门红包预警群。

命中条件建议：

1. 单用户 1 小时内发送达到返奖门槛红包次数 > 30
2. 单用户 1 小时内达标红包总金额 > 3,000,000
3. 同一房间 10 分钟内 >= 10 笔 B 档红包，进入观察名单，只告警不直接拦截
4. 同一房间 10 分钟内 >= 20 笔 B 档红包，且满足下面任一附加条件时，判为高风险：
   - 房间自然礼物 / 语音互动明显偏低，但红包领完率异常高
   - 自然互动热度 : 红包领取热度 <= 3 : 7
   - 最近多笔红包有效领取用户高度重复
   - 命中同设备 / 同 IP / 关联账号互抢

命中处理：

1. 观察阈值：发送飞书预警，房间进入观察名单
2. 强风险阈值：发送飞书预警，相关发送人进入限制级黑名单
3. 强风险命中后，后续该用户发出的达标红包不参与返奖
4. 记录风控日志，支持人工复核和解除限制

飞书提醒字段：

- 预警级别：观察 / 强风险
- 房间 ID、发送人 UID、红包 ID
- 近 1 小时达标红包次数与总金额
- 近 10 分钟 B 档红包笔数
- 红包领完率、自然礼物 / 语音互动指标
- 命中设备 / IP / 关联账号摘要
- 后台审核链接

### 规则 3：高风险房间识别

命中条件建议：

1. 房间长期只有固定极少数用户互抢
2. 房间红包领完率异常高但自然礼物行为极低
3. 房间红包中奖率显著高于正常区间

命中后：

1. 发送飞书预警提醒
2. 房间进入高风险房间名单
3. 该房间内发送红包的用户进入限制级黑名单
4. 后续该房间的达标红包可配置为仅告警、限制返奖或关闭返奖资格

### 规则 4：日返奖上限

平台维度：

- 全站每日返奖预算上限：200 美金，即 2,000,000 金币
- 触顶后自动关闭返奖
- 熔断后新领完的红包不再开奖，记录为 `blocked_by_budget`
- 已经开奖成功但发奖失败的记录仍进入补发队列，不受后续熔断重复影响

### 规则 5：发奖幂等

返奖只允许结算一次。

服务端要求：

1. 以红包 ID 为唯一开奖主键
2. 开奖结果落库
3. 重试不重复开奖
4. 发奖失败进入补发队列
5. 补发成功后写入钱包事务 ID，禁止二次补发

### 规则 6：生产环境模拟器限制

生产环境中，模拟器、压测工具、内部测试账号和灰度验证账号不得参与真实返奖结算。

建议规则：

1. 识别模拟器设备指纹、自动化框架特征、异常设备参数和内部测试账号标签
2. 命中模拟器特征的账号默认进入危险级，禁止发红包和领红包
3. 生产验证必须使用白名单房间 / 白名单账号 / 沙箱币种，不参与真实返奖预算
4. 模拟器命中日志单独归档，避免和普通用户风险日志混淆
5. 若模拟器账号已经产生真实红包或返奖记录，自动标记为待人工复核

### 规则 7：预防机器人自动抢红包

场景：

脚本监听红包 IM 消息，毫秒级自动调用领取接口，导致正常用户难以抢到红包。

建议规则：

1. 红包发出后设置服务端随机延迟窗口，例如 1 - 3 秒后才允许领取
2. 服务端返回 `claim_not_before`，领取接口必须校验时间窗口，客户端倒计时只做展示
3. 领取动作增加行为验证，例如必须点击动画中的按钮并携带一次性行为 token
4. 行为 token 由服务端签发和校验，不能被绕过客户端直接调用接口复用
5. 监控领取响应时间：红包发出到领取 < 500ms 的账号标记为可疑
6. 单用户连续 N 个红包领取响应时间都 < 1s 时，触发验证码或临时冷却；建议 N 默认 3，可后台配置
7. 可疑账号进入观察名单；持续命中或叠加设备 / IP 风险时升级限制级或危险级

### 规则 8：基础开奖反作弊

1. 发送人自领不计入有效领取人数
2. 红包必须在 3 分钟内领完才开奖
3. 有效领取人数至少 2 人
4. 红包过期未领完时，剩余金额返还发送人，本次不触发返奖抽奖

## 6.4 同房间 B 档红包聚集风险说明

“同房间短时集中出现异常多笔 B 档红包”的风险主要在于：

1. 平台补贴被短时间集中消耗
   - B 档返奖比例最高，若短时间连续触发，会快速放大补贴支出
2. 团伙套利风险高
   - 可能出现固定一批账号轮流发 B 档红包、互相抢完来制造开奖条件
3. 伪造房间繁荣
   - 房间看起来红包活跃很高，但并不是自然社交消费，会误导运营判断
4. 交互噪音变大
   - 连续多笔 B 档红包会带来大量悬浮窗、结果提醒和开奖记录，影响正常体验

## 7. 红包领取流程优化

新版 PDF 已有基础领取流程，但还缺 3 个关键体验补丁：

1. 用户中奖返奖怎么提醒
2. 返奖弹窗和红包领取弹窗冲突怎么办
3. 红包过多时，弹窗和领取队列怎么控

下面是建议优化方案。

## 7.1 用户中奖返奖怎么提醒

### 场景 A：发送人在线，且当前没有强弹窗

表现建议：

1. 红包被领完后，先展示“红包结果弹窗”
2. 若该红包命中返奖，则在红包结果弹窗内直接合并返奖信息

推荐文案：

`你的红包已被抢完，并触发返奖，1,000 金币已返还至钱包`

这样可以避免：

1. 先看一个“红包领完”弹窗
2. 再看一个“返奖成功”弹窗

导致连续两次打断。

### 场景 B：发送人在线，但当前已有高优先级弹窗

例如：

1. 充值弹窗
2. 离开房间确认弹窗
3. 更高优先级活动弹窗

处理方式建议：

1. 先发一个弱提醒 toast：
   - `你的红包触发返奖，奖励已到账`
2. 同时在右上角或钱包角标增加红点
3. 当前强弹窗关闭后，再补展示返奖结果卡片
4. 若 30 秒内仍未可展示，则降级进入“消息中心 / 红包记录”

### 场景 C：发送人不在当前房间 / 已离开

处理方式建议：

1. 不强弹模态窗
2. 使用：
   - 系统消息
   - 钱包到账通知
   - 红包记录页红点

推荐文案：

`你在房间 880321 发送的红包已触发返奖，1,000 金币已返还`

## 7.2 弹窗冲突解决方案

### 问题一：发送方结果弹窗 vs 返奖成功弹窗

建议：

`合并为一个结果弹窗`

弹窗结构如下：

1. 顶部：红包已领完
2. 中部：领取明细
3. 底部：
   - 若未中奖：展示“本次未触发返奖”
   - 若中奖：展示“返奖 xxx 金币已到账”

这样一张弹窗就覆盖结果 + 返奖。

### 问题二：自己抢自己红包时，领取结果弹窗 vs 发送方结果弹窗

新版 PDF 已经说明：

若自己可以领自己发的红包，且只剩一份时，只展示一个结果弹窗。

建议进一步明确：

1. 自己抢到最后一份红包
2. 若该红包同时满足返奖并中奖
3. 则展示一个“合并结果弹窗”

内容包括：

- 你抢到了自己的红包
- 红包已被抢完
- 本次返奖结果

避免三连弹窗。

### 问题三：充值弹窗 / 离房确认弹窗 / 活动弹窗同时出现

建议建立统一弹窗优先级：

#### S 级

1. 充值相关阻断弹窗
2. 离开房间确认弹窗
3. 安全风控类强提示

#### A 级

1. 返奖中奖结果弹窗
2. 红包领完结果弹窗

#### B 级

1. 红包领取弹窗
2. 多红包选择弹窗

#### C 级

1. toast
2. 轻提示条
3. 红点提醒

规则：

1. 低优先级不得打断高优先级
2. 同一红包产生的多个 A/B 弹窗，优先合并
3. 超过排队上限后，降级为 toast 或消息中心

## 7.3 红包过多时的队列优化

### 问题

当房间内同时存在多个待领取红包时，可能出现：

1. 用户不停点开红包
2. 多个领取弹窗连续出现
3. 结果弹窗、返奖结果、横幅、飘屏叠加，导致信息拥挤

### 优化方案

#### 方案 1：领取入口只保留一个主悬浮窗

现有规则已经有“右侧红包悬浮窗 + 数量徽标”。

建议继续强化：

1. 始终只展示一个主悬浮入口
2. 徽标显示待领取红包数量
3. 倒计时显示最早过期的红包

#### 方案 2：点击后先进入红包列表，而不是连续弹单个红包

当待领取红包数量 > 1 时：

1. 先打开“红包列表底部弹层”
2. 列表排序建议：
   - 即将过期优先
   - 返奖资格红包优先
   - 高金额优先
   - 最新发送优先
3. 用户点选具体红包后再进入领取弹窗

#### 方案 3：弹窗队列上限

建议设置：

- 模态弹窗排队上限：3

超过 3 个后：

1. 新的低优先级弹窗不再进入模态队列
2. 自动降级为：
   - toast
   - 列表状态更新
   - 消息中心记录

#### 方案 4：同一红包同一角色只保留一个结果态

示例：

发送人对同一红包不应出现：

1. 红包领完弹窗
2. 返奖中奖弹窗
3. 钱包到账弹窗

而应合并成：

`红包最终结果卡`

#### 方案 5：设置“结果摘要卡”

当短时间内多个红包都已结束时，可展示一张摘要卡：

示例：

`你有 3 个红包已完成，1 个中奖返奖，2 个未中奖`

点击后跳到开奖记录列表。

## 13. 最终建议结论

### 13.1 返奖规则

采用新版文档中的两档返奖：

- A 档：20k - 49,999，3% 返 5%
- B 档：>= 50k，5% 返 10%

### 13.2 风控核心

上线必须同时具备：

1. 红包领完后开奖
2. 有效领取人数 >= 2
3. 发送人自领不计入有效领取人数
4. 同设备 / 同 IP 异常拦截
5. 红包黑名单等级
6. 关联设备 / IP 同步限制返奖资格
7. 高频大额红包和高风险房间飞书预警
8. 平台每日返奖预算上限 2,000,000 金币
9. 发奖幂等与补发队列
10. 生产环境模拟器限制
11. 防脚本抢红包的随机延迟、行为 token、响应时延监控

### 13.3 领取流程优化核心

建议强制落地：

1. 发送方红包结果弹窗与返奖结果合并
2. 多红包场景优先进入红包列表，而非连续弹窗
3. 模态弹窗队列设置上限 3
4. 超限后自动降级为 toast / 红点 / 消息中心

### 13.4 后台重点

后台不仅要配返奖门槛和概率，更要能看：

1. A/B 档效果
2. 风控命中
3. 红包黑名单和关联设备 / IP
4. 飞书预警发送与处理状态
5. 弹窗冲突与到达率
6. 返奖补贴率是否失控
